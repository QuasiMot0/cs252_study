## What source Does

The `source` builtin (also spelled `.` in POSIX shells) reads commands from a file and executes them in the **current shell process** — not in a child. This means any state changes (environment variables set, directory changes) take effect in the running shell. For example:

```bash
source ~/.bashrc       # re-read and re-execute the startup file
```

## The Problem: Flex Has an Internal Buffer

The Flex-generated scanner does not read one character at a time directly from `yyin`. Instead, it reads in large chunks into an internal buffer called a **`YY_BUFFER_STATE`**. The scanner's `yylex()` function then consumes characters from this buffer.

When implementing `source`, a naive approach would be:

```c
yyin = fopen(filename, "r");   // redirect lex's input file
```

**This is wrong.** Flex has already read characters from the old input into its internal buffer. Reassigning `yyin` does not flush this buffer. The scanner will continue consuming characters from the old buffer first, possibly losing the switch entirely or mixing up the two inputs.

**Exam question (lab3_quiz Q17):** Changing `yyin` alone when implementing `source` loses the buffered data in the current `YY_BUFFER_STATE`. You must use `yy_switch_to_buffer()`.

## Correct Approach: YY_BUFFER_STATE Switching

Flex provides functions to explicitly manage buffer states:

```c
// Save the current buffer state
YY_BUFFER_STATE saved = YY_CURRENT_BUFFER;

// Open the file to source
FILE *f = fopen(filename, "r");
if (f == NULL) { perror("source"); return; }

// Create a new buffer for the file and switch to it
YY_BUFFER_STATE new_buf = yy_create_buffer(f, YY_BUF_SIZE);
yy_switch_to_buffer(new_buf);

// The scanner now reads from the file
// When yylex() returns EOF (end of file), the sourcing is done.

// Restore the original buffer
yy_switch_to_buffer(saved);
fclose(f);
```

### What Each Function Does

| Function | Purpose |
|---|---|
| `YY_CURRENT_BUFFER` | Macro that evaluates to the currently active `YY_BUFFER_STATE` |
| `yy_create_buffer(file, size)` | Allocate a new `YY_BUFFER_STATE` backed by `file` |
| `yy_switch_to_buffer(buf)` | Make `buf` the active buffer; the scanner reads from it next |
| `yy_delete_buffer(buf)` | Free a `YY_BUFFER_STATE` when done |

## Why source Is a Builtin (Not Forked)

Like `cd` and `setenv`, `source` must run in the shell process:
- If it ran in a child, any `setenv`, `cd`, or other state changes in the sourced file would disappear when the child exits.
- The whole point of `source` is to modify the current shell's environment.

## Key Terms

| Term | Meaning |
|---|---|
| `source` | Builtin that executes commands from a file in the current shell process |
| `YY_BUFFER_STATE` | Flex's internal object holding the buffered input for one input source |
| `yy_switch_to_buffer(buf)` | Switch the Flex scanner to read from a different `YY_BUFFER_STATE` |
| `yy_create_buffer(f, size)` | Create a new `YY_BUFFER_STATE` backed by file `f` |
| `YY_CURRENT_BUFFER` | Macro returning the currently active `YY_BUFFER_STATE` |
