## What Flex Does

Flex (Fast Lexical Analyzer) generates a C scanner from a set of pattern-action rules written in `shell.l`. The scanner reads the raw character stream one character at a time and groups characters into *tokens* — the meaningful units that Yacc sees. The shell's Flex file defines all the token types the grammar needs to recognize.

## Token Types in shell.l

The shell scanner recognizes at least the following tokens:

| Token name | Characters matched | Meaning |
|---|---|---|
| `WORD` | any unquoted string of characters | a command name or argument |
| `GREAT` | `>` | redirect stdout (truncate) |
| `GREATGREAT` | `>>` | redirect stdout (append) |
| `LESS` | `<` | redirect stdin |
| `TWOGREAT` | `2>` | redirect stderr |
| `GREATAMPERSAND` | `>&` | redirect stdout+stderr to file |
| `PIPE` | `\|` | pipe between commands |
| `AMPERSAND` | `&` | background execution |
| `NEWLINE` | `\n` | end of command line |

## The Longest-Match Rule

When more than one rule could match the current input, Flex always applies the rule that matches the **longest** string. This is critical for `>>` vs `>`:

- The scanner sees `>>`
- Rule for `GREAT` matches one character: `>`
- Rule for `GREATGREAT` matches two characters: `>>`
- Flex picks `GREATGREAT` because it is longer

**Exam question (lab3_quiz Q10):** If the input is `>>`, Flex fires the `GREATGREAT` rule, not two separate `GREAT` rules. This is the longest-match rule in action.

## Tie-breaking: First Rule Wins

When two rules match the same-length string, Flex fires the rule that appears **first** in `shell.l`. This means the ordering of rules in the file matters for disambiguation.

## How the Scanner Feeds the Parser

The Flex scanner is called by Yacc via `yylex()`. Each call to `yylex()` returns the numeric constant for the next token (e.g., `GREAT`, `WORD`) and optionally sets `yylval` with associated data (e.g., the string value of a `WORD` token). This token stream is what the Yacc grammar rules match against.

## Common Pitfall

```c
/* This would be wrong — '>' would always fire before '>>' could match */
">"   { return GREAT; }
">>"  { return GREATGREAT; }
```

Because Flex uses longest-match, the order above actually works correctly — `>>` will still match `GREATGREAT`. But if both rules matched the same length, you would need to worry about ordering.

## Key Terms

| Term | Meaning |
|---|---|
| Flex | Tool that generates a C scanner from pattern-action rules |
| Token | A categorized unit of input (WORD, GREAT, PIPE, etc.) |
| Longest-match | Flex rule: always fire the rule matching the most characters |
| `yylex()` | The function Flex generates; called by Yacc to get the next token |
| `yylval` | Global variable where the scanner stores a token's associated value |
