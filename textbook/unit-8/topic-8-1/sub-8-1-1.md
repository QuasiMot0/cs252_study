## The Shell's Place in UNIX

The shell sits between the user and the UNIX operating system. The system is organized in layers: the OS kernel at the bottom, the standard library (libc) in the middle, and shell programs above that. When you type a command, the shell interprets it and makes the appropriate system calls.

There are many shell implementations — `/bin/sh`, `/bin/csh`, `/bin/tcsh`, `/bin/bash`, `zsh` — but all share the same fundamental three-part design.

## The Three-Part Architecture

```
User Input
    │
    ▼
┌─────────────────────────────────────────────┐
│                  PARSER                     │
│   Flex scanner + Yacc/Bison grammar         │
│   Reads tokens, builds a parse tree         │
└─────────────────────┬───────────────────────┘
                      │  parse tree
                      ▼
┌─────────────────────────────────────────────┐
│                 EXECUTOR                    │
│   Walks the parse tree; calls               │
│   fork / exec / dup2 / pipe / waitpid       │
└─────────────────────┬───────────────────────┘
                      │  delegates expansion
                      ▼
┌─────────────────────────────────────────────┐
│               SUBSYSTEMS                    │
│   • Wildcard/glob expansion                 │
│   • Subshell / backtick expansion           │
│   • Environment variable expansion ($VAR)   │
│   • Tilde (~) expansion                     │
│   • Built-in commands (cd, setenv, exit…)  │
└─────────────────────────────────────────────┘
```

### Parser

The parser reads what the user typed and turns it into a structured parse tree the executor can act on. It has two components:

- **Flex** (the scanner/lexer) breaks the raw character stream into *tokens* — the smallest meaningful units such as a word, a `|`, a `>`, or an `&`.
- **Yacc/Bison** (the parser) takes the stream of tokens and matches them against grammar rules, building a *parse tree* that captures the hierarchical structure of the command.

The parser never executes anything; it only builds the tree.

### Executor

The executor traverses the parse tree and makes things happen. For each command node in the tree it:

1. Sets up I/O redirection using `dup2`.
2. Creates a child process using `fork`.
3. Runs the external program in the child using `execvp`.
4. Connects adjacent commands in a pipeline using `pipe`.
5. Waits for foreground commands to finish using `waitpid`.

### Subsystems

Subsystems are helper modules invoked during parsing or execution to handle special shell features:

| Subsystem | What it does |
|---|---|
| Wildcard expansion | Expands `*.c` into a sorted list of matching filenames via `readdir` + regex |
| Subshell expansion | Runs `` `cmd` `` or `$(cmd)` in a child shell process and substitutes its stdout output |
| Environment variable expansion | Replaces `$VAR` with the variable's value from the environment |
| Tilde expansion | Replaces leading `~` with the `HOME` environment variable value |
| Built-in commands | Runs `cd`, `setenv`, `unsetenv`, `exit` directly in the shell process (not forked) |

## Why This Decomposition?

Separating the three concerns makes each piece simpler:

- The **parser** knows nothing about processes or file descriptors — it only understands syntax.
- The **executor** knows nothing about token matching — it only knows how to wire up processes.
- The **subsystems** each handle one specific expansion and can be tested independently.

## Key Terms

| Term | Meaning |
|---|---|
| Token | Smallest lexical unit (WORD, PIPE, GREAT, etc.) |
| Parse tree | Tree structure representing a parsed command line |
| Fork | System call to create a child process |
| Exec | System call to replace a process's image with a new program |
| Builtin | A command the shell executes directly without forking |
