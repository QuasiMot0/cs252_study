## Why Start Conditions Are Needed

The Flex scanner is normally in a single default state (`INITIAL`). But parsing a shell command line requires context-sensitive behavior: once you recognize that you are inside a word (a sequence of non-delimiter characters), you want different rules to apply than when you are between words. Flex **start conditions** let you switch between named scanner states.

In the shell lexer, `WORD_STATE` is the start condition used when the scanner is in the middle of building up a multi-character token such as a command name or argument.

## How WORD_STATE Works

In `INITIAL` state, when the scanner sees the first character of a word, it:

1. Switches to `WORD_STATE` using `BEGIN(WORD_STATE)`.
2. Starts accumulating characters.

While in `WORD_STATE`:
- More word characters are appended to the current token.
- A delimiter character (space, `|`, `>`, `&`, newline, etc.) signals the end of the word.

When a delimiter is encountered in `WORD_STATE`, the scanner needs to:
1. Finalize and return the accumulated WORD token.
2. Push the delimiter character back into the input so that the `INITIAL` state can process it as its own token on the next call.

## unput(): Pushing Characters Back

The `unput(c)` function pushes the character `c` back into the Flex input buffer. The next call to the scanner will read `c` first before continuing with the remaining input.

**Why this matters:** When in `WORD_STATE` and a `|` is seen, the `|` ends the word, but the `|` itself needs to be tokenized as a `PIPE` token. Rather than handling `|` explicitly in `WORD_STATE`, the scanner pushes it back:

```c
/* In WORD_STATE, hitting a delimiter: */
unput(yytext[0]);   /* push the delimiter back */
BEGIN(INITIAL);     /* return to INITIAL to process it */
return WORD;        /* return the accumulated word */
```

**Exam question (lab3_quiz Q19):** When the scanner hits a delimiter like `|` while in `WORD_STATE`, calling `unput(yytext[0])` pushes that delimiter back so that the `INITIAL` state can re-process it as a `PIPE` or other token.

## myunputc(): The Reverse-Order Trick

Subshell expansion uses a related function called `myunputc()`, defined at the top of `shell.l`. It pushes a single character into Flex's internal lookahead buffer. Since Flex's buffer is **LIFO** (last in, first out), to inject a string `"hello"` into the scanner input you must push the characters in **reverse order**: first `'o'`, then `'l'`, `'l'`, `'e'`, `'h'`.

**Exam question (lab3_quiz Q13):** `myunputc` is called in reverse order because the lex buffer is LIFO — the last character pushed will be the first one read.

## YY_BUFFER_STATE: Switching Input Sources

For the `source` builtin (which reads commands from a file), simply redirecting `yyin` to the new file is not enough — Flex has already buffered characters from the old input into an internal `YY_BUFFER_STATE` object, and those buffered characters would be lost.

The correct approach:
1. Save the current buffer state with `yy_create_buffer` / `YY_CURRENT_BUFFER`.
2. Open the new file and create a new buffer with `yy_create_buffer(file, YY_BUF_SIZE)`.
3. Switch to the new buffer with `yy_switch_to_buffer(new_buf)`.
4. When the file is exhausted, restore the saved buffer with `yy_switch_to_buffer(saved_buf)`.

**Exam question (lab3_quiz Q17):** Changing `yyin` alone when implementing `source` loses the buffered data in the current `YY_BUFFER_STATE`. You must use `yy_switch_to_buffer()`.

## Key Terms

| Term | Meaning |
|---|---|
| Start condition | A named scanner state in Flex; switch with `BEGIN(state)` |
| `INITIAL` | The default start condition |
| `WORD_STATE` | Start condition active while building a multi-character WORD token |
| `unput(c)` | Push character `c` back into Flex's input stream |
| `myunputc(c)` | Shell-specific helper to inject a character into the lex lookahead buffer |
| `YY_BUFFER_STATE` | Flex's internal object holding buffered input; must be saved/restored to switch input sources |
