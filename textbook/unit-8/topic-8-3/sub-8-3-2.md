## What exec Does

The `exec` family of system calls **replaces** the current process's program image with a new one. The process ID stays the same, but code, data, and stack are replaced by the new program. After a successful `exec`, the original code is gone — the new program runs in its place.

## execvp()

The shell uses `execvp()` because it:
- Searches the `PATH` environment variable to find the executable (the `p` in `execvp`).
- Accepts arguments as a vector (the `v` in `execvp`).

```c
int execvp(const char *file, char *const argv[]);
```

- `file`: the name (or path) of the executable.
- `argv`: a NULL-terminated array of `char*` argument strings. `argv[0]` must be the program name itself (by convention).

## Argument Array Format

`argv` must look like this:

```
argv[0] → "ls"
argv[1] → "-al"
argv[2] → "/tmp"
argv[3] → NULL        ← mandatory NULL terminator
```

**The array must be terminated with a NULL pointer.** Without it, `execvp` does not know where the argument list ends.

## Converting C++ Strings to char* for execvp

The CS252 shell stores arguments in `std::string` objects, but `execvp` requires `char*`. The conversion:

```cpp
// Collect args into a char* array
char **args = new char*[_numOfArguments + 1];  // +1 for NULL
for (int i = 0; i < _numOfArguments; i++) {
    args[i] = (char *) _arguments[i]->c_str();
}
args[_numOfArguments] = NULL;   // NULL terminator

execvp(args[0], args);
```

## When Code After exec() Runs

`execvp` **never returns** on success — the current process image has been replaced. Code after `execvp` only runs if `execvp` fails (e.g., the executable was not found).

```c
execvp(args[0], args);
// If we reach here, exec FAILED
perror("execvp");
_exit(1);
```

**Exam question (lab3_quiz Q8):** The code after `execvp` runs only on failure. If `exec` succeeded, that code is unreachable.

## Why _exit() After Failed exec

After `fork()`, the child process has a copy of the parent's stdio buffers (because `fork` copies the entire address space). If the child calls `exit()`, it flushes those buffers — potentially producing duplicate output on top of whatever the parent writes later.

`_exit()` terminates immediately without flushing stdio buffers, avoiding this duplication. Always use `_exit()` in the child after a failed `exec`.

## Worked Example: Running `ls -al`

```c
pid_t ret = fork();
if (ret == 0) {
    // Child
    char *args[] = { "ls", "-al", NULL };
    execvp("ls", args);
    perror("execvp");   // only reached on failure
    _exit(1);
}
// Parent continues here...
waitpid(ret, NULL, 0);
```

## Key Terms

| Term | Meaning |
|---|---|
| `execvp` | Replaces current process with a new program; searches PATH |
| `argv[0]` | Must be the program name by convention |
| NULL terminator | Required last element of the `argv` array |
| `_exit()` | Terminates immediately; does not flush stdio buffers |
| `exit()` | Terminates and flushes stdio buffers — dangerous to call in child after fork |
