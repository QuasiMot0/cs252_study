## What fork() Does

`fork()` creates an exact copy of the calling process. After `fork()` returns, there are two processes: the **parent** (the original caller) and the **child** (the new copy). Both processes continue executing from the instruction immediately after the `fork()` call.

## Return Values

The return value is how you tell which process you are in:

| Return value | Who sees it | Meaning |
|---|---|---|
| `0` | child process | "I am the child" |
| Positive integer | parent process | The child's PID |
| `-1` | parent process | `fork()` failed; child was never created |

```c
pid_t ret = fork();
if (ret == -1) {
    perror("fork");
    exit(1);
}
if (ret == 0) {
    // child process
} else {
    // parent process; ret == child's PID
}
```

## What Is Copied vs. What Is Shared

After `fork()`, the child gets a **copy** of the parent's state, but some things are genuinely shared:

### Copied (child gets its own independent version)

- **Virtual address space** (code, data, stack, heap) — but see Copy-on-Write below
- **File descriptor table** — the child's table is a separate copy; closing a descriptor in the child does not close it in the parent, and vice versa

### Shared (child and parent both reference the same underlying object)

- **Open file table entries** — even though the child has its own fd table, the entries in that table point to the same underlying open file objects in the kernel. This means the child and parent **share the current file position** for any file that was open before `fork()`.

## Copy-on-Write (COW)

Actually copying the entire address space on every `fork()` would be very slow. Modern kernels use **copy-on-write**: after `fork()`, the parent and child share the same physical memory pages, marked read-only. The first time either process writes to a page, the kernel makes a private copy of that page for the writer. This makes `fork()` fast when the child immediately calls `exec()`.

## Reference Counting on File Objects

Each open file table entry has a **reference count** tracking how many file descriptors in how many processes point to it. When `fork()` copies the fd table, the reference counts are incremented. When `close()` is called on an fd, the count is decremented. The file object is freed only when the count reaches zero.

This is why both parent and child must close unused pipe ends: until both close the write end, the read end will never see EOF.

## Exam Trace Example (lab3_quiz Q1)

```c
ret = fork();           // (A) — both parent and child execute from here
if (ret == 0) {
    execvp(…);          // (B) — only child; replaces child's process image
    printf("D\n");      // never reached if exec succeeds
}
printf("C\n");          // (C) — only parent (child was replaced by exec)
```

Execution order: parent prints `C`; child executes `B` (exec); `D` is never reached.

## Common Pitfall

```c
pid_t ret = fork();
if (ret == 0) {
    // WRONG: don't call exit() after failed exec in a child
    // exit() flushes stdio buffers — those buffers were also copied from the parent
    // and could produce duplicate output
    execvp(args[0], args);
    exit(1);     // use _exit(1) instead
}
```

Use `_exit()` (not `exit()`) in the child after a failed `exec`, because `_exit()` terminates immediately without flushing stdio buffers that may have been copied from the parent.

## Key Terms

| Term | Meaning |
|---|---|
| `fork()` | System call that creates a copy of the calling process |
| PID | Process ID; unique identifier for each process |
| Copy-on-Write | Optimization where memory pages are shared until one process writes to them |
| Open file table | Kernel structure tracking open files; shared across `fork()` |
| Reference count | Number of fds (across all processes) pointing to an open file object |
| `_exit()` | Terminates immediately without flushing stdio buffers |
