## Why the Shell Must Wait

After `fork()`, the child process eventually exits. Until the parent calls `wait()` or `waitpid()`, the child remains in the process table as a **zombie** — it has released most of its resources, but its exit status is kept for the parent to collect. The shell must call one of the wait functions to reap each child and collect its exit status.

## wait()

```c
pid_t wait(int *wstatus);
```

- Blocks until **any** child exits.
- Returns the PID of the child that exited.
- Stores the child's exit information in `wstatus`.

`wait()` is simple but too coarse for a real shell: you cannot choose which child to wait for, and you cannot ask it not to block.

## waitpid()

```c
pid_t waitpid(pid_t pid, int *wstatus, int options);
```

- `pid = -1`: wait for any child (same as `wait`).
- `pid > 0`: wait specifically for the child with that PID.
- `options = 0`: block until the specified child exits.
- `options = WNOHANG`: return immediately if no child has exited yet (return value 0 means no child was ready).

The shell uses `waitpid` for foreground commands and `waitpid(-1, &status, WNOHANG)` inside the `SIGCHLD` handler to reap background children without blocking.

## WEXITSTATUS: Extracting the Exit Code

The `wstatus` value is not the raw exit code — it encodes multiple pieces of information about how the process terminated. The `WEXITSTATUS` macro extracts the actual exit code:

```c
int wstatus;
waitpid(child_pid, &wstatus, 0);
int exit_code = WEXITSTATUS(wstatus);   // e.g., 0 for success, non-zero for failure
```

`WEXITSTATUS` is only valid if `WIFEXITED(wstatus)` is true (i.e., the process exited normally rather than being killed by a signal).

## WNOHANG in Signal Handlers

**Exam question (lab3_quiz Q7):** Inside a `SIGCHLD` signal handler, you must not block. Using `waitpid(-1, &wstatus, WNOHANG)` returns immediately if no child is ready, rather than blocking the shell. The canonical pattern is:

```c
void sigchld_handler(int sig) {
    int wstatus;
    while (waitpid(-1, &wstatus, WNOHANG) > 0) {
        // child was reaped; optionally record its exit status
    }
}
```

The `while` loop is necessary because one `SIGCHLD` signal may represent multiple children that exited at nearly the same time — signals are not queued, so a single delivery may correspond to more than one child exit.

## Zombie Processes

**Exam question (lab3_quiz Q2):** When a child exits, the kernel sends `SIGCHLD` to the parent and keeps the child as a zombie in the process table until the parent calls wait/waitpid. A zombie consumes a process table slot but has no memory or CPU resources.

Shells deal with zombie accumulation from background processes by installing a `SIGCHLD` handler that calls `waitpid(-1, …, WNOHANG)` in a loop.

## Key Terms

| Term | Meaning |
|---|---|
| Zombie | A process that has exited but has not yet been reaped by its parent |
| `wait()` | Block until any child exits; return its PID |
| `waitpid()` | Wait for a specific child (or any child with `pid = -1`) with optional flags |
| `WNOHANG` | Option flag: return immediately if no child has exited |
| `WEXITSTATUS(s)` | Macro to extract the exit code from a `wstatus` value |
| `SIGCHLD` | Signal sent to parent when a child process exits |
