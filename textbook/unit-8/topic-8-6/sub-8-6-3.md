## Foreground vs. Background

When the user appends `&` to a command, the shell grammar sets the `background` flag on the `Command` object. The executor checks this flag after forking the child:

```c
if (background == false) {
    // Foreground: shell waits for the command to finish
    waitpid(child_pid, &wstatus, 0);
} else {
    // Background: shell does NOT wait; returns to prompt immediately
    // The child will be reaped later by the SIGCHLD handler
}
```

**Exam question (lab3_quiz Q12):** When `background = true`, the parent skips `waitpid` and returns immediately to print the prompt. The `SIGCHLD` handler (which calls `waitpid(-1, &wstatus, WNOHANG)`) reaps the background child when it eventually exits.

## Why Skipping waitpid Is Safe

The shell has a `SIGCHLD` handler installed that runs `waitpid(-1, &wstatus, WNOHANG)` in a loop whenever any child exits. This handler reaps both:
- Background processes the shell did not wait for
- Processes that exit inside a pipeline (pipe processes are also reaped via `SIGCHLD`)

Without the handler, background processes would pile up as zombies.

## Interactive vs. Non-Interactive Mode: isatty()

The shell should print a prompt (`myshell>`) only when running interactively — reading commands from a terminal. In non-interactive mode (e.g., when the shell is reading from a script file or from a pipe in a test harness), printing the prompt would corrupt the output.

```c
if (isatty(0)) {
    // stdin is a terminal — print the prompt
    prompt();
}
```

`isatty(fd)` returns non-zero if `fd` refers to a terminal device, and zero if it refers to a file, pipe, or socket.

**Exam question (lab3_quiz Q16):** `isatty(0)` returns 0 when stdin is redirected from a file or pipe. This is exactly the condition under which the test scripts run the shell — so the prompt check prevents spurious output in test output.

## The Full Background Flow

```
User types: sleep 10 &

1. Parser recognizes AMPERSAND → sets background = true
2. Executor forks child
3. Child: execvp("sleep", ["sleep", "10", NULL])
4. Parent: skips waitpid, prints prompt, reads next command

... 10 seconds later ...

5. sleep exits → kernel sends SIGCHLD to parent
6. SIGCHLD handler: waitpid(-1, &wstatus, WNOHANG) reaps sleep
```

## Key Terms

| Term | Meaning |
|---|---|
| Foreground | Shell blocks with `waitpid` until the command finishes |
| Background | Shell does not wait; command runs concurrently with the shell |
| `isatty(fd)` | Returns non-zero if `fd` is a terminal; zero if it's a file or pipe |
| `SIGCHLD` handler | Reaps background children with `waitpid(-1, &s, WNOHANG)` |
