## What Is a Zombie?

When a process exits, the kernel does not immediately remove it from the process table. Instead, it releases the process's memory and resources but keeps a small record — the process's PID, exit status, and accounting information — until the parent collects it by calling `wait()` or `waitpid()`.

This "waiting to be reaped" state is called a **zombie** process (shown as `<defunct>` in `ps` output).

A zombie:
- Has no code, data, or stack in memory
- Still occupies a slot in the process table
- Cannot be killed with `SIGKILL` — it's already dead, just uncollected

## Why Zombies Exist

The design is intentional: the parent may need to inspect the child's exit status after the child has already exited. If the kernel immediately destroyed all trace of the child, the parent would have no way to get the exit code. The zombie state is the kernel saying "I have the exit status ready; come get it."

## SIGCHLD: The Notification Mechanism

When any child exits, the kernel automatically sends `SIGCHLD` to the parent process.

**Exam question (lab3_quiz Q2):** A process that exits and becomes a zombie also causes `SIGCHLD` to be sent to its parent.

## The SIGCHLD Handler Pattern

For foreground commands, the shell calls `waitpid(child_pid, …, 0)` immediately after forking, so foreground zombies are reaped right away. For background commands (and for processes that exit inside a pipe), the shell installs a `SIGCHLD` handler:

```c
void sigchld_handler(int sig) {
    int wstatus;
    while (waitpid(-1, &wstatus, WNOHANG) > 0) {
        // A child was reaped; optionally print notification
    }
}
```

Key points:
- `waitpid(-1, …)` — wait for **any** child
- `WNOHANG` — return immediately if no child is ready; never block inside a signal handler
- The **`while` loop** is essential

**Why a while loop?** Signals are not queued one-for-one. If three background processes exit simultaneously, the kernel may deliver only one `SIGCHLD`. The while loop keeps calling `waitpid` until no more zombies remain, ensuring all three are reaped from a single handler invocation.

**Exam (slides):** Your shell gets a `SIGCHLD` signal for every process that exits inside a PIPE and for background processes. Use the while loop pattern above.

## When Zombies Accumulate

If a shell runs many background commands without a `SIGCHLD` handler, zombies pile up. Each zombie consumes a process table entry. On Linux, there is a system-wide limit on the number of processes; enough zombies can exhaust it and prevent any new processes from being created.

## Key Terms

| Term | Meaning |
|---|---|
| Zombie | A process that has exited but not yet been reaped by its parent |
| `SIGCHLD` | Signal sent to the parent when a child exits |
| `waitpid(-1, &s, WNOHANG)` | Reap any available zombie without blocking |
| Reaped | A child whose exit status has been collected by the parent via wait/waitpid |
| Process table | Kernel data structure that tracks all processes; zombies occupy a slot here |
