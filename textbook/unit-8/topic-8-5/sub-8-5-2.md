## The Multi-Command Pipeline Problem

For a pipeline like `ls -al | grep foo | sort | head -5`, the shell must:

1. Create one pipe between each adjacent pair of commands.
2. Fork a child for each command.
3. Wire up each child's stdin and stdout to the appropriate pipe ends.
4. Close all unused pipe ends in both parent and children.
5. Wait for all children to finish.

## Shell Execution Strategy

The executor saves the shell's original stdin and stdout before starting, then loops over each command in the pipeline:

```c
// Save shell's own stdin/stdout so we can restore them afterward
int tmpin  = dup(0);
int tmpout = dup(1);

int fdin = tmpin;   // first command reads from saved stdin (or input file)

for each command i in pipeline {
    // Set up stdin for this command
    dup2(fdin, 0);
    close(fdin);

    if (i is NOT the last command) {
        // Create a pipe for this command's stdout
        int fdpipe[2];
        pipe(fdpipe);
        dup2(fdpipe[1], 1);    // stdout → pipe write end
        close(fdpipe[1]);
        fdin = fdpipe[0];      // next command reads from this pipe
    } else {
        // Last command: stdout goes to saved stdout or output file
        dup2(tmpout, 1);
    }

    // Fork and exec
    pid_t ret = fork();
    if (ret == 0) {
        execvp(args[0], args);
        _exit(1);
    }
    // Parent loops to next command
}

// Restore shell's stdin/stdout
dup2(tmpin,  0); close(tmpin);
dup2(tmpout, 1); close(tmpout);

// Wait for all foreground children
for each child PID:
    waitpid(pid, &status, 0);
```

## The prevPipe Pattern

**Exam question (lab3_quiz Q20):** `prevPipe` stores the **read end** (`fd[0]`) of the pipe created for the previous command in the loop. It becomes the stdin source for the next command. On each iteration:

- `fdin` = `prevPipe` (the read end of the pipe the previous command writes into)
- The current command creates a new pipe if it is not the last command
- The new pipe's read end becomes the next `prevPipe`

## Why the Parent Must Close Pipe Write Ends

After forking a child that uses `fdpipe[1]` as its stdout, the parent no longer needs `fdpipe[1]`. The parent must close it:

- If the parent keeps `fdpipe[1]` open, then when the child exits, the read end (`fdpipe[0]`) will **not** see EOF — the parent's open copy of the write end prevents it.
- The next command waiting to read from `fdpipe[0]` will block forever.

## Worked Example: `ls -al | grep foo`

```
Before loop: fdin = tmpin (terminal stdin)

Iteration 1 (ls -al):
  dup2(tmpin, 0)       ← ls reads from terminal stdin
  pipe(fdpipe)         ← create pipe for ls's output
  dup2(fdpipe[1], 1)   ← ls's stdout goes to pipe write end
  close(fdpipe[1])     ← close redundant write end in parent
  fdin = fdpipe[0]     ← grep will read from pipe read end
  fork → exec ls -al

Iteration 2 (grep foo):
  dup2(fdpipe[0], 0)   ← grep reads from pipe read end
  close(fdpipe[0])
  dup2(tmpout, 1)      ← grep's stdout goes to terminal (last command)
  fork → exec grep foo

After loop:
  dup2(tmpin, 0); close(tmpin)
  dup2(tmpout, 1); close(tmpout)
  waitpid for ls, then grep
```

## Key Terms

| Term | Meaning |
|---|---|
| `prevPipe` | The read end of the previous pipe; becomes stdin for the next command |
| `fdin` | The file descriptor to use as stdin for the current command in the loop |
| `fdout` | The file descriptor to use as stdout for the current command |
| `tmpin` / `tmpout` | Saved copies of the shell's original stdin/stdout |
