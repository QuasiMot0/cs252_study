## The Blocking Rule for Pipes

A `read()` on the read end of a pipe blocks until one of two things happens:

1. **Data is available** in the pipe buffer.
2. **All file descriptors referencing the write end** of the pipe are closed — at which point `read()` returns 0 (EOF).

If any process anywhere in the system still holds the write end open — even if it never writes anything — the reader will block forever.

## The Deadlock Scenario

**Exam question (lab3_quiz Q4):** Consider this pattern:

```c
int fd[2];
pipe(fd);

pid_t ret = fork();
if (ret == 0) {
    // Child: write "hello" to pipe
    close(fd[0]);
    write(fd[1], "hello", 5);
    close(fd[1]);
    exit(0);
}

// Parent: attempt to read TWICE
close(fd[1]);           // ← parent closes its write end (correct)
char buf[100];

read(fd[0], buf, 5);    // reads "hello" → OK
read(fd[0], buf, 5);    // ← BLOCKS FOREVER if parent forgot to close fd[1]
```

Wait — in the code above the parent does close `fd[1]` before reading. So the second `read` returns 0 (EOF) because the only writer (the child) has exited and closed `fd[1]`.

**The deadlock version** — parent forgets to close `fd[1]`:

```c
// WRONG: parent never closes fd[1]
read(fd[0], buf, 5);    // reads "hello"
read(fd[0], buf, 5);    // BLOCKS FOREVER
                        // The parent itself still has fd[1] open,
                        // so the pipe never sends EOF
```

The parent holds the write end open (fd[1]). The child exits and closes its copy. But the parent's fd[1] is still alive. So the pipe has one remaining writer (the parent itself) — but the parent is blocked reading from fd[0]. Deadlock.

## Shell Implications

In a multi-command pipeline, the shell parent process creates pipes and then forks children. After forking each child, the parent must close the write end of every pipe it is done with:

```c
pipe(fdpipe);
fork();   // child gets copies of fdpipe[0] and fdpipe[1]

// Parent MUST close fdpipe[1] after child is forked
close(fdpipe[1]);   // Without this, the next command's read() will block forever
```

If the shell has N commands and N-1 pipes, it must close N-1 write-end copies in the parent.

## Closing the Read End Too

The writer should also close the read end it does not use. If the pipe buffer fills up and the write end is still open, `write()` blocks waiting for the reader to drain the buffer. A writer that keeps the read end open is holding a reference that could prevent EOF from being signaled on the read side for other processes — and more importantly, it is wasting a file descriptor.

## Key Terms

| Term | Meaning |
|---|---|
| Pipe buffer | Kernel memory holding data written to a pipe but not yet read |
| EOF on pipe | Returned by `read()` when all write-end references are closed |
| Deadlock | State where two processes block waiting for each other — neither can proceed |
| Dangling write end | A write-end fd that is never closed; prevents EOF from reaching the reader |
