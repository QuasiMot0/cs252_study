## What a Parse Tree Represents

When Yacc reduces grammar rules, the C action code embedded in each rule builds a data structure — the **parse tree** — that represents the structure of the command the user typed. The executor then walks this tree to actually run the command.

## Example Parse Trees

### Single command: `a b`

```
command_line
    └── pipe_list
            └── cmd_and_args
                    ├── command: "a"
                    └── args: ["a", "b"]
```

`a` is the program name; `b` is the single argument.

### Two arguments: `ls a b`

```
command_line
    └── pipe_list
            └── cmd_and_args
                    ├── command: "ls"
                    └── args: ["ls", "a", "b"]
```

Note that the command name (`ls`) is also included in the argument array — `execvp` requires `argv[0]` to be the program name.

### Pipeline with redirection: `ls a b | sort > out`

```
command_line
    ├── pipe_list
    │       ├── cmd_and_args[0]
    │       │       ├── command: "ls"
    │       │       └── args: ["ls", "a", "b"]
    │       └── cmd_and_args[1]
    │               ├── command: "sort"
    │               └── args: ["sort"]
    └── io_modifier_list
            └── io_modifier: GREAT "out"
```

The executor sees two commands to connect with a pipe, and after the last command redirects its stdout to the file `out`.

## How the Executor Walks the Tree

The executor's `execute()` method (typically in `Command.cc`) iterates over the list of `SimpleCommand` objects in the pipeline. For each command:

1. **Set up stdin (`fdin`):** For the first command, `fdin` is either the default stdin or an input-redirected file. For subsequent commands, `fdin` is the read end (`fd[0]`) of the pipe created by the previous iteration.

2. **Set up stdout (`fdout`):** For all commands except the last, `fdout` is the write end (`fd[1]`) of a newly created pipe. For the last command, `fdout` is either the default stdout or an output-redirected file.

3. **Fork a child.**

4. **In the child:** use `dup2` to wire `fdin` → fd 0 (stdin) and `fdout` → fd 1 (stdout), then call `execvp`.

5. **In the parent:** close the write end of each pipe after forking the child that uses it as stdout, to allow EOF propagation.

6. **After all forks:** restore the shell's own stdin/stdout (using saved copies made with `dup` before the loop), then `waitpid` for all foreground children.

## The `_currentSimpleCommand` Global

During parsing, Yacc action code builds a `SimpleCommand` object and appends it to the current `Command` object. The global `Command::_currentSimpleCommand` pointer tracks the command being built. Each time the `cmd_and_args` rule is reduced, a new `SimpleCommand` is created and attached to the pipeline.

## Key Terms

| Term | Meaning |
|---|---|
| Parse tree | Tree structure produced by Yacc; input to the executor |
| `SimpleCommand` | Object holding one command's name and argument list |
| `Command` | Object holding the full pipeline: list of `SimpleCommand` objects plus I/O redirections |
| `fdin` | File descriptor used as stdin for the current command in the pipeline loop |
| `fdout` | File descriptor used as stdout for the current command in the pipeline loop |
