## What Yacc Does

Yacc (Yet Another Compiler Compiler) — or its modern replacement Bison — takes a grammar written in `shell.y` and generates a C parser. The parser reads the token stream produced by Flex and matches it against grammar rules. When a rule is fully matched, Yacc executes the C action code embedded in the rule, which typically builds nodes in a parse tree.

## The Shell Grammar Hierarchy

The shell grammar is composed of the following non-terminals, from smallest to largest:

```
arg_list
cmd_and_args
pipe_list
io_modifier
io_modifier_list
background_optional
command_line
command_list
```

Each level handles one specific aspect of the command syntax.

### arg_list

An `arg_list` is a list of one or more words that follow the command name:

```yacc
arg_list : arg_list WORD   { /* add WORD to current command's argument list */ }
         | WORD             { /* start argument list with WORD */ }
         ;
```

### cmd_and_args

A single command with its arguments:

```yacc
cmd_and_args : WORD arg_list  { /* command with arguments */ }
             | WORD            { /* command with no arguments */ }
             ;
```

### pipe_list

A chain of commands separated by `|`. The rule is **left-recursive**, which is how Yacc handles a sequence of arbitrary length:

```yacc
pipe_list : pipe_list PIPE cmd_and_args  { /* append command to pipeline */ }
          | cmd_and_args                  { /* start new pipeline */ }
          ;
```

**Exam question (lab3_quiz Q11):** The recursive `pipe_list` grammar rule is what allows the shell to handle pipelines of any number of commands. One rule handles both the two-command and N-command cases.

### io_modifier

A single I/O redirection specification:

```yacc
io_modifier : GREAT WORD           { /* redirect stdout to file (truncate) */ }
            | GREATGREAT WORD      { /* redirect stdout to file (append) */ }
            | LESS WORD            { /* redirect stdin from file */ }
            | TWOGREAT WORD        { /* redirect stderr to file */ }
            | GREATAMPERSAND WORD  { /* redirect stdout and stderr to file */ }
            ;
```

### io_modifier_list

Zero or more I/O redirections:

```yacc
io_modifier_list : io_modifier_list io_modifier  { /* add redirection */ }
                 | /* empty */
                 ;
```

### background_optional

Whether the command runs in the background:

```yacc
background_optional : AMPERSAND  { /* set background flag to true */ }
                    | /* empty */  { /* foreground (default) */ }
                    ;
```

### command_line

A complete command with optional I/O and background:

```yacc
command_line : pipe_list io_modifier_list background_optional NEWLINE
             | NEWLINE   { /* empty/blank line */ }
             ;
```

### command_list

A sequence of command lines:

```yacc
command_list : command_list command_line  { /* accumulate commands */ }
             | /* empty */
             ;
```

## Grammar Execution Flow

When the user types `ls -al | sort > out`, the parse proceeds:

1. `ls` → `WORD` → starts `cmd_and_args`
2. `-al` → `WORD` → added to `arg_list`
3. `|` → `PIPE` → triggers `pipe_list` recursive rule
4. `sort` → `WORD` → new `cmd_and_args`
5. `>` → `GREAT`, `out` → `WORD` → matched by `io_modifier`
6. `\n` → `NEWLINE` → completes `command_line`

## Key Terms

| Term | Meaning |
|---|---|
| Non-terminal | A grammar symbol that can be expanded into other symbols |
| Left recursion | A rule of the form `A : A x` — standard technique in Yacc for lists |
| Action code | C code inside `{ … }` in a Yacc rule; runs when the rule is matched |
| `$$` | In Yacc actions, the value returned by the current rule |
| `$1`, `$2`, … | Values of the symbols on the right-hand side of the rule |
