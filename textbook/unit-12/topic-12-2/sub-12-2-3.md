# Multi-Table Joins via WHERE Clause

## The Problem: Data Spread Across Tables

Relational databases avoid redundancy by splitting data into separate tables. To answer questions that span multiple tables, you need to **join** them.

Example schema:
- `employees(id, name, dept_id, salary)`
- `departments(id, name, building)`

## Implicit Join Syntax (Old Style)

List multiple tables in `FROM`, separated by commas. Then use `WHERE` to specify the join condition:

```sql
SELECT e.name, d.name
FROM employees e, departments d
WHERE e.dept_id = d.id;
```

This returns each employee's name alongside their department's name.

## Table Aliases

`FROM employees e` assigns the alias `e` to the `employees` table for this query. You then use `e.name` instead of `employees.name`. This is especially useful when joining tables with overlapping column names.

## The Cartesian Product Trap

**Without** the WHERE join condition, you get every possible combination of rows — called a **Cartesian product** (or cross join):

```sql
-- DANGEROUS: No join condition
SELECT e.name, d.name
FROM employees e, departments d;
-- If employees has 50 rows and departments has 5 rows → 250 result rows
-- Every employee is paired with every department regardless of actual relationship
```

Always include the join condition. The Cartesian product is almost never what you want.

## Joining Three Tables

For n tables, you need n-1 join conditions:

```sql
SELECT s.name, c.title, e.grade
FROM students s, courses c, enrollments e
WHERE e.student_id = s.id
  AND e.course_id = c.id;
```

- `enrollments` is the junction table linking students to courses
- Two WHERE conditions link the three tables

## Adding Regular Filters

You can combine join conditions with regular filters using AND:

```sql
SELECT e.name, d.name
FROM employees e, departments d
WHERE e.dept_id = d.id
  AND e.salary > 60000;
-- Returns high-earning employees with their department name
```

## Silent Logical Errors

If you have a typo in the join condition, SQL won't necessarily give you an error:

```sql
-- Typo: e.dept_di instead of e.dept_id
WHERE e.dept_di = d.id
-- If dept_di doesn't exist: column not found error (caught)
-- If dept_di exists but holds wrong values: wrong pairings (silent bug!)
```

Always verify your results make sense.

## Key Takeaways

| Concept | Rule |
|---------|------|
| Implicit join | `FROM t1, t2 WHERE t1.fk = t2.pk` |
| Table alias | `FROM employees e` — use `e.col` in rest of query |
| Cartesian product | What you get if you forget the WHERE join condition |
| n tables | Need n-1 join conditions |
| Combined filter | Use AND to add regular filters alongside join conditions |
