# Relational Databases: Tables, Rows, Columns, and Keys

## What is a Relational Database?

A **relational database** organizes data into **tables** (also called *relations*). Each table represents one type of entity (e.g., employees, products, orders). Tables consist of:

- **Rows** (tuples/records): one instance of the entity
- **Columns** (attributes/fields): one property of the entity, with a declared data type

All rows in a relational table are unordered — the database makes no guarantees about row order unless you explicitly ask for it with `ORDER BY`.

## Primary Keys

A **primary key** is a column (or combination of columns) that **uniquely identifies each row** in a table. Rules:

- No two rows can have the same primary key value
- A primary key column cannot be NULL
- Common choices: an auto-incrementing integer `id`, a username, an email address

Example — `employees` table:

| id (PK) | name    | dept | salary |
|---------|---------|------|--------|
| 1       | Alice   | CS   | 75000  |
| 2       | Bob     | HR   | 55000  |
| 3       | Carol   | CS   | 82000  |

Here `id` is the primary key.

## Foreign Keys

A **foreign key** is a column in one table that references the primary key of another table. This creates a link between the two tables.

Example — `orders` table with a foreign key to `customers`:

| order_id (PK) | customer_id (FK) | total |
|---------------|-----------------|-------|
| 101           | 5               | 49.99 |
| 102           | 5               | 12.00 |
| 103           | 7               | 99.00 |

`customer_id` here references `id` in the `customers` table. Each order is linked to the customer who placed it.

## Referential Integrity

**Referential integrity** is the rule that every foreign key value must match an existing primary key in the referenced table. The database enforces this to prevent orphaned records:

- You **cannot** insert an order with `customer_id = 99` if customer 99 doesn't exist
- You **cannot** delete a customer who still has orders (unless you delete the orders first, or use CASCADE)

## Why "Relational"?

The term comes from mathematical **set theory** and **relational algebra** (E.F. Codd, 1970). Each table is a *relation* — a set of tuples sharing the same attributes. The power of relational databases comes from combining tables using joins to answer complex queries without duplicating data.

## Key Takeaways

| Concept | Definition |
|---------|-----------|
| Table | Set of rows sharing the same columns |
| Row | One record/instance |
| Column | One attribute with a declared type |
| Primary Key | Uniquely identifies each row; not NULL |
| Foreign Key | References a primary key in another table |
| Referential Integrity | FK must match an existing PK |
