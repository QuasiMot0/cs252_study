## Process Scheduling Overview

From the slides:
> "From the user's point of view, an OS allows running multiple processes simultaneously."
> "In reality, the OS runs one process after another to give the illusion that multiple processes run simultaneously."
> "The Process Scheduler is the OS subsystem that runs one process after the other and decides what process to run next."
> "A Context Switch is the procedure used by the OS to switch from one process to another"

## Process States

From the slides:

| State | Meaning |
|---|---|
| **New** | Process is being initialized |
| **Ready** | Candidate to run; another process is currently using the CPU |
| **Running** | Currently using the CPU; at most one per CPU |
| **Waiting** | Waiting for an event (I/O, interrupt) |
| **Terminated** | Process is exiting |

Transitions: New → Ready → Running ↔ Ready; Running → Waiting → Ready; Running → Terminated.

## I/O Bound vs CPU Bound Processes

From the slides:
- **I/O bound**: most of the time waiting for events (mouse, keyboard, network). Only in ready/running briefly. **Most processes are I/O bound.**
- **CPU bound**: need the CPU for long periods (scientific computation, video rendering, compilers). Mostly in ready or running state.
- **Most applications are I/O bound.**

## Context Switch Steps

From the slides:
1. Save current running process in process table (PC, registers, stack)
2. Load next ready process from process table and put registers and PC in the CPU

Context switches happen when:
- A process goes to wait state (I/O or event)
- A process yields the CPU voluntarily
- A timer interrupt fires (preemptive scheduling only)

## Types of Scheduling

From the slides:
> "There are two types of scheduling: Non Preemptive Scheduling, Preemptive Scheduling."

### Non-Preemptive Scheduling (Cooperative)

From the slides:
> "In Non Preemptive Scheduling a context switch happens only when the running process goes to a waiting state or when the process gives up the CPU voluntarily (no timer context switch)."
> "This is a very primitive type of scheduling. It is also called cooperative scheduling."
> "It was used in Windows 3.1 and initial versions of MacOS."
> "The main problem of Non Preemptive Scheduling is that a misbehaved process that loops forever may hold the CPU and prevent other processes from running."

Advantages: more control, simple to implement.

### Preemptive Scheduling

From the slides:
> "In Preemptive Scheduling a context switch happens periodically (every 1/100sec or quantum time) as a result of a timer interrupt."
> "A timer interrupt will cause a context switch, that is, the running process to go to ready state at the end of the ready-queue and the process that has been the longest in ready state at the front of the ready-queue will go to running state."
> "Preemptive scheduling is implemented in UNIX, and Windows 95 and above."

Advantages: robust (no process can monopolize CPU); fairness (OS ensures equal CPU usage).

## Round Robin Scheduling

From the slides:
> "Ready processes are in a ready queue."
> "Every time that a time quantum expires, a process is dequeued from the front of the ready queue and put in running state"
> "The previous running process is added to the end of the ready queue."

### Round Robin Example (T=10ms)

| Process | Burst Time |
|---|---|
| p1 | 24ms |
| p2 | 3ms |
| p3 | 3ms |

Execution order: P1(10) → P2(3) → P3(3) → P1(10) → P1(4)

Completion times: p1=30ms, p2=13ms, p3=16ms

**Average completion time = (13 + 16 + 30) / 3 = 19.6ms**

## Multilevel Feedback-Queue Scheduling

From the slides:
> "Instead of a having a single queue of ready processes, there are multiple queues of different priorities."
> "The scheduler will schedule the ready processes with the highest priority first."
> "Within processes of the same priority round robin is used."

Problem: low priority processes may never run if higher priority processes always exist.

Solution — **Process Aging**:
> "The longer a process stays in a queue, the priority is artificially increased. The process will be scheduled at some point. After that, the priority is reset to the initial priority."

Priority feedback rule:
> "The smaller the CPU burst estimate, the higher the priority. The larger the CPU burst estimate the lower the priority."

Interactive (I/O-bound) processes that finish their burst before the quantum expires get their priority increased. CPU-bound processes that use the full quantum have their priority decreased.

## Process Table

From the slides, each process table entry contains:
- PID, Command and args, Environment Vars, Current Dir, Owner (User ID)
- Stdin/Stdout/Stderr, List of memory mappings, List of open files
- Per thread: PC (Program Counter), Registers, Stack, State (Wait/Ready/Running)

The process table entry contains enough information to restart (put in running state) a process that is waiting or in ready state.

## Key Terms

| Term | Meaning |
|---|---|
| Process Scheduler | OS subsystem that decides which process runs next |
| Context switch | Save current process state; load next process state |
| Non-preemptive (cooperative) | Context switch only on wait or voluntary yield; used in Windows 3.1 |
| Preemptive | Context switch driven by timer interrupt (quantum); used in UNIX, Windows 95+ |
| Round Robin | FIFO ready queue; each process gets one quantum then goes to end of queue |
| Time quantum | Duration of one round-robin time slice; standard = 10ms (1/100 sec) |
| Multilevel Feedback Queue | Multiple priority queues; interactive processes get higher priority |
| Process Aging | Increasing a waiting process's priority over time to prevent starvation |
| CPU burst | Short period when an I/O-bound process needs the CPU |
| I/O bound | Process mostly in waiting state; most applications are I/O bound |
| CPU bound | Process mostly in ready/running state; needs CPU for long periods |
