## What Is an Interrupt?

From the slides:
> "An interrupt is an event that requires immediate attention. In hardware, a device sets the interrupt line to high."
> "When an interrupt is received, the CPU will stop whatever it is doing and it will jump to the 'interrupt handler' that handles that specific interrupt."
> "After executing the handler, it will return to the same place where the interrupt happened and the program continues."

Examples: move mouse, type key, ethernet packet arrives.

## Steps of Servicing an Interrupt

From the slides:
1. The CPU saves the **Program Counter and registers** in the execution stack.
2. CPU looks up the corresponding interrupt handler in the **interrupt vector**.
3. CPU jumps to interrupt handler and runs it.
4. CPU restores the registers and returns back to the place in the program that was interrupted. The program continues execution as if nothing happened.
5. In some cases it retries the instruction that was interrupted (e.g., Virtual memory page fault handlers).

## Running with Interrupts (Asynchronous I/O)

From the slides:
> "Interrupts allow CPU and device to run in parallel without waiting for each other."

Sequence:
1. OS requests device operation (e.g., write to disk)
2. Device runs the operation; OS does other work in parallel
3. When operation is complete, device interrupts the OS
4. OS services the interrupt and continues

## Polling (Synchronous I/O)

From the slides:
> "Alternatively, the OS may decide not use interrupts for some devices and wait in a busy loop until completion."

```
OS requests Device operation
While request is not complete
  do nothing;
Continue execution.
```

> "This type of processing is called 'polling' or 'busy waiting' and wastes a lot of CPU cycles."
> "Polling is used for example to print debug messages in the kernel (kprintf)."

- **Polling** = **Synchronous Processing** (device and program execute synchronously)
- **Interrupt** = **Asynchronous Processing** (device and CPU run in parallel)

## Interrupt Vector

From the slides:
> "It is an array of pointers that point to the different interrupt handlers of the different types of interrupts."

Entries include:
- Hard Drive Interrupt handler
- USB Interrupt handler (mouse, keyboard)
- Ethernet Card Interrupt handler
- Page Fault Interrupt handler

The interrupt vector is initialized at **boot time** and modified only when drivers are added. It can only be modified in **kernel mode** (security).

## Interrupts and Kernel Mode

From the slides:
> "Interrupts run in kernel mode. Why?"
> "An interrupt handler must read device/CPU registers and execute instructions only available in kernel mode."

## Types of Interrupts

From the slides:

1. **Device Interrupts** — generated by devices when a request is complete or an event happens:
   - mouse moved, key typed, Ethernet packet arrives, hard drive read/write complete, CD inserted

2. **Math Exceptions** — generated by the CPU on math errors:
   - divide by zero

3. **Page Faults** — generated by the MMU (Memory Management Unit):
   - Invalid address → SEGV signal to the process
   - Valid address but page not in memory → CPU loads page from disk
   - Invalid permission (writing to read-only page) → SEGV signal

4. **Software Interrupt** — generated by software with a special assembly instruction:
   - This is how a program in user mode requests OS services (system calls)

## Key Terms

| Term | Meaning |
|---|---|
| Interrupt | Event requiring immediate CPU attention; device sets interrupt line high |
| Interrupt handler (ISR) | Kernel function that services a specific interrupt |
| Interrupt vector | Array of pointers to interrupt handlers, indexed by interrupt type |
| Polling / busy waiting | Synchronous loop waiting for device completion; wastes CPU cycles |
| Synchronous processing | CPU waits for device (polling) |
| Asynchronous processing | CPU and device run in parallel (interrupt-driven) |
| Device interrupt | Hardware-generated interrupt from mouse, keyboard, disk, etc. |
| Page fault | MMU-generated interrupt on invalid/missing memory access |
| Software interrupt | Assembly instruction that triggers a kernel mode switch (used by syscalls) |
