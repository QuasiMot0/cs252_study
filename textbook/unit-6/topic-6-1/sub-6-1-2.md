## Directories and Hard Links

From the slides:
> "A directory is a file that contains a list of pairs (file name, I-node number)."
> "Each pair is also called a hard-link."

A directory stores mappings from names to inode numbers — it does not store the file data itself. Multiple directory entries can map to the same inode number; these are called **hard links**.

## Reference Counting

From the slides:
> "The reference count in the I-node keeps track of the number of directories where the I-node appears."
> "When you remove a file from a directory, the reference count is decreased."
> "When the reference-count reaches 0, the file is removed from the disk."

When you `rm` a file:
1. The directory entry (name → inode) is removed.
2. The inode's reference count is decremented.
3. If the count reaches **0**, the data blocks are freed and the inode is released.
4. If other hard links to the same inode still exist, the data remains.

In some OSes, the reference count is also incremented while the file is open (to prevent deletion of a file that is currently being read).

## Creating a Hard Link

```bash
ln target-file name-link
```

This creates a new directory entry `name-link` pointing to the same inode as `target-file`. After this command, both names refer to exactly the same file — the same data and the same inode.

## Hard Link Limitation

From the slides:
> "Hard Links cannot cross partitions, that is, a directory cannot list an I-node of a different partition."

Because inode numbers are only unique within a partition, hard links cannot span partitions or file systems.

## Soft Links (Symbolic Links)

From the slides:
> "A soft-link is a pair of the form (file name, i-node-number-with-file-storing-path)."
> "Soft-links can point to files in different partitions."
> "A soft-link does not keep a reference count to track of the target file."
> "If the target file is removed, the symbolic link becomes invalid (dangling symbolic link)."

```bash
ln -s target-file name-link    # create a symbolic link
```

Soft links (symlinks) store a **path** rather than a direct inode reference. They can cross partitions but become dangling if the target is removed.

## Hard vs Soft Link Comparison

| Property | Hard Link | Soft Link |
|---|---|---|
| Storage | Inode number in directory | Path string in a special inode |
| Can cross partitions? | No | Yes |
| Reference counted? | Yes — prevents deletion | No — can dangle |
| Survives target removal? | Yes (is the file) | No (becomes invalid) |
| Create with | `ln target link` | `ln -s target link` |

## Key Terms

| Term | Meaning |
|---|---|
| Directory | File storing (name, inode number) pairs |
| Hard link | Directory entry pointing to an existing inode |
| Reference count | Number of hard links to an inode; file deleted when count reaches 0 |
| Soft link / Symlink | Directory entry storing a path; can cross partitions; may dangle |
| `ln target link` | Create hard link |
| `ln -s target link` | Create symbolic (soft) link |
| Dangling symlink | Symbolic link whose target has been removed |
