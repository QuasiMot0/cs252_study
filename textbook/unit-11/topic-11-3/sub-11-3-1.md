## What Is HTTP?

From the slides:

> "HTTP (HyperText Transfer Protocol) is a request-response protocol. A client sends a request to a server, and the server replies with either the requested file or an error message."

- HTTP is an **application-layer** protocol used to distribute web content.
- **Client/server model**: server sends objects in response to requests; client requests, receives, and displays objects.
- HTTP is a **stateless** protocol — the server does not remember previous client requests.

## GET Request Format

From the slides, a client retrieves a file by sending a GET request in this format:

```
GET <sp> <Document Requested> <sp> HTTP/1.1 <crlf>
{<Other Header Information> <crlf>}*
<crlf>
```

**Breaking it down:**

- `<sp>`: a single whitespace character
- `<Document Requested>`: the file path being requested — e.g., `/index.html`, or just `/` for the default file
- `<crlf>`: carriage return (ASCII 13) followed by linefeed (ASCII 10), written as `\r\n`
- `{<Other Header Information> <crlf>}*`: optional header lines (zero or more)
- The final **blank line** (`<crlf>` on its own) signals the **end of the request**

**Example:**

```
GET /index.html HTTP/1.1\r\n
Host: example.com\r\n
\r\n
```

## HTTP Request Methods

From the slides:

- **GET**: requests should only retrieve data and should have no other effect.
- **POST**: process the request according to the posted message.
- Other methods: HEAD, PUT, DELETE, CONNECT, TRACE, OPTIONS, PATCH

## Server Processing

From the slides, once the server receives a GET request it:

1. Parses the request to extract the file path.
2. Searches for the requested file.
3. Sends back one of two responses depending on whether the file was found.

## Success Response (200 OK)

From the slides:

```
HTTP/1.1 <sp> 200 <sp> Document follows <crlf>
Server: <sp> <Server-Type> <crlf>
Content-type: <sp> <Document-Type> <crlf>
{<Other Header Information> <crlf>}*
<crlf>
<Document Data>
```

**Breaking it down:**

- `HTTP/1.1 200 Document follows`: status line confirming success
- `Server`: identifies the server (e.g., "CS 252 lab5")
- `Content-type`: tells the client what kind of document is being sent — e.g., `text/html`, `image/gif`, `text/plain`
- Final `<crlf>` on its own: blank line separating headers from body
- `<Document Data>`: the actual file content

## Not-Found Response (404)

From the slides:

```
HTTP/1.1 404 File Not Found\r\n
Content-type: text/plain\r\n
\r\n
Could not find the URL.
```

## Example Exchange (via telnet)

From the slides (HTTP Protocol Example):

```
telnet google.com 80

GET / HTTP/1.1

HTTP/1.1 200 OK
Content-Type: text/html; charset=ISO-8859-1
<other headers>
<message body>
```

## Basic Server Algorithm

From the slides (Basic Server):

- Open a passive (listening) socket on port 80, 443, or another port.
- Loop forever:
  - Accept new TCP connection
  - Read request from TCP connection and parse it
  - Frame the appropriate response header (200 OK or 404 Not Found)
  - Write the response header to the TCP connection
  - Write the requested document to the TCP connection
  - Close the TCP connection (HTTP/1.1)

## Key Terms

| Term | Meaning |
|---|---|
| HTTP | HyperText Transfer Protocol; stateless request-response protocol |
| GET | HTTP method to retrieve a resource; no side effects |
| `<crlf>` | Carriage return + linefeed (`\r\n`); line terminator in HTTP |
| Blank line | A lone `<crlf>` that signals end of HTTP headers |
| `Content-type` | Header specifying the MIME type of the response body |
| Status line | First line of HTTP response: version + code + reason phrase |
| Stateless | Server does not retain state between requests |
