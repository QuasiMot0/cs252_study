## Purpose of Basic HTTP Authentication

From the slides:

> "Background: to protect the private resource on a dedicated server."

The authentication scheme used is **RFC 7617** — "Basic HTTP Authentication." It prevents unauthorized users from accessing protected resources, but **does not encrypt** the communication — credentials are sent in encoded (not encrypted) text.

From the lab handout:
> "Basic HTTP Authentication alone is not truly secure. Since our server uses plain HTTP rather than HTTPS, all traffic travels unencrypted, meaning anyone sniffing the network can intercept the Authorization header and instantly decode credentials."
> "Use TLS to protect the security and privacy of the data transfer."

## The Two-Round Exchange

From the slides:

### Round 1 — Client Request Without Credentials

```
GET /index.html HTTP/1.1\r\n
\r\n
```

The server checks for an `Authorization` header. It is absent, so the server responds:

```
HTTP/1.1 401 Unauthorized\r\n
WWW-Authenticate: Basic realm="myhttpd-cs252"\r\n
\r\n
<html>401 Unauthorized</html>
```

The browser displays a login prompt to the user.

### Round 2 — Client Request With Credentials

The user enters their username and password. The browser encodes them as `username:password` in **Base64** and resends the request:

```
GET /index.html HTTP/1.1\r\n
Authorization: Basic Y3MyNTI6cGFzc3dvcmQ=\r\n
\r\n
```

`Y3MyNTI6cGFzc3dvcmQ=` is the Base64 encoding of `cs252:password`.

The server decodes the `Authorization` header, verifies the credentials, and if valid:

```
HTTP/1.1 200 Document follows\r\n
...
<html>Body</html>
```

## How the Server Checks Credentials

From the lab handout:

1. **No Authorization header present**: respond with `401 Unauthorized` + `WWW-Authenticate: Basic realm="<something>"`.
2. **Authorization header present**: decode the Base64 value, compare against stored credentials.
   - If **valid**: serve the requested file normally.
   - If **invalid**: return `401 Unauthorized` again.

All subsequent requests from the same browser session include the `Authorization` header automatically — the user only types credentials once.

## Setting Up Credentials

From the lab handout: create a credentials file containing `username:password` with no trailing newline, then Base64-encode it:

```bash
cat mycredentials.txt | base64
```

- Do **not** use `cs252` or `password` as credentials.
- Do **not** use your Purdue career account credentials.

## Key Terms

| Term | Meaning |
|---|---|
| Basic HTTP Authentication | RFC 7617; credentials sent as Base64-encoded `username:password` |
| Base64 | Encoding scheme that converts binary/text to ASCII-safe characters |
| `WWW-Authenticate` | Server response header requesting authentication; includes realm |
| `Authorization` | Client request header carrying `Basic <base64-encoded-credentials>` |
| realm | A string identifying the protection space; shown to the user in the login prompt |
| 401 Unauthorized | HTTP status code meaning authentication is required or credentials are invalid |
| TLS | Transport Layer Security; needed to actually encrypt credentials in transit |
