## HTTP Versions and Connection Persistence

From the slides:

| Version | Year | Connection Model |
|---|---|---|
| HTTP/1.0 | 1996 | **Non-persistent** TCP connection |
| HTTP/1.1 | 1997 | Keep-alive feature allowing a **persistent** TCP connection |
| HTTP/2.0 | â€” | Extends persistent connection by **multiplexing** many concurrent requests/responses through a single TCP/IP connection |
| HTTP/3.0 | â€” | Uses **UDP/QUIC** to improve communication speed and avoid TCP/IP connection congestion (head-of-line blocking) |

## Non-Persistent vs Persistent Connections

From the slides:

| | Non-Persistent | Persistent |
|---|---|---|
| 1 | Open TCP connection | Open TCP connection |
| 2 | At most **one object** is sent through the TCP connection | **Multiple objects** are sent through the TCP connection |
| 3 | Close TCP connection | Close TCP connection |

**Persistent connections require fewer RTTs** â€” the TCP handshake cost is paid once, then reused for multiple requests.

## Non-Persistent Example

From the slides:

1. HTTP Client **initiates** TCP connection (SYN).
2. HTTP server **accepts**, notifies the client (SYN+ACK).
3. HTTP Client sends a HTTP request for `index.html` (ACK).
4. HTTP server sends HTTP response with `index.html`.
5. HTTP server **closes** TCP connection.
6. HTTP client receives response, parses `index.html`, finds 1 image â†’ **repeats steps 1â€“5** for the image.

Each object (html file, image) requires a separate TCP connection.

## Persistent Example

From the slides:

1â€“3. Same TCP handshake as non-persistent.
4. HTTP server sends response with `index.html`. **Connection stays open.**
5. HTTP client receives response, finds 1 image.
6. HTTP client sends HTTP request for `image.jpeg` **over the same connection**.
7. HTTP server sends response with `image.jpeg`.
8. HTTP server closes TCP connection.

One TCP connection serves all objects on the page.

## Content-Type Detection by File Extension

From the lab handout: the `Content-type` header tells the browser what type of document is being sent. Content-type is determined by the file extension:

| File Extension | Content-Type |
|---|---|
| `.html`, `.htm` | `text/html` |
| `.gif` | `image/gif` |
| `.jpg`, `.jpeg` | `image/jpeg` |
| `.txt` | `text/plain` |
| (not found, 404) | `text/plain` |

From the lab handout:
> "Content-type: tells the client what kind of document is being sent. There are many types available."
> - `text/html`: an HTML page
> - `image/gif`: a GIF image
> - `text/plain`: plain text, also present when HTTP status is 404

## File Path Resolution

From the lab handout: once authenticated, the server may serve any document â€” not just `index.html`. Path resolution rules:

| Request Path | Returned File (relative to http-root-dir) |
|---|---|
| `/` | `/htdocs/index.html` |
| `/simple.html` | `/htdocs/simple.html` |
| `/icons/ball.gif` | `/icons/ball.gif` |
| `/htdocs/logo.svg` | `/htdocs/logo.svg` |
| `/logo.svg` | `/htdocs/logo.svg` |

If the request path does not include a subfolder, look for the file inside `htdocs/`. Otherwise, leave the path unchanged.

## Key Terms

| Term | Meaning |
|---|---|
| Non-persistent | New TCP connection per HTTP object; more RTTs |
| Persistent | Single TCP connection reused for multiple HTTP objects; fewer RTTs |
| Keep-alive | HTTP/1.1 feature enabling persistent connections |
| Multiplexing | HTTP/2 feature; multiple requests/responses interleaved over one TCP connection |
| QUIC | UDP-based transport used by HTTP/3; avoids head-of-line blocking |
| Content-type | MIME type header; tells browser how to interpret the response body |
| Head-of-line blocking | One slow request stalls all subsequent requests in a TCP connection |
