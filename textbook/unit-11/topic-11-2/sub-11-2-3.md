## Types of Server Concurrency

From the slides, there are five server models:

1. **Iterative Server** — one client at a time
2. **Fork Process After Request** — new process per connection
3. **Create New Thread After Request** — new thread per connection
4. **Pool of Threads** — fixed set of threads sharing the listening socket
5. **Pool of Processes** — fixed set of processes sharing the listening socket

## 1. Iterative Server

From the slides:

```c
void iterativeServer(int serverSocket) {
    while (1) {
        int clientSocket = accept(serverSocket, &sockInfo, &alen);
        if (clientSocket >= 0) {
            dispatchHTTP(clientSocket);
        }
    }
}
```

- `dispatchHTTP` itself closes `clientSocket` when done.
- **Not concurrent** — handles one client at a time; all other clients queue while one is being served.
- Simple but creates a bottleneck under load.

The Lab 5 server uses this as the default mode (no flag passed).

## 2. Fork Process After Request (`-f`)

From the slides:

```c
void forkServer(int serverSocket) {
    while (1) {
        int clientSocket = accept(serverSocket, &sockInfo, &alen);
        if (clientSocket >= 0) {
            int ret = fork();
            if (ret == 0) {
                dispatchHTTP(clientSocket);
                exit(0);
            }
            close(clientSocket);
        }
    }
}
```

- Parent accepts the connection, forks a child to handle it, then immediately closes its copy of `clientSocket` and loops back to `accept()`.
- Child handles the request and exits.
- Must deal with **zombie processes**: when the child exits, the parent must eventually `wait()` or use `SIGCHLD` handling to reap child processes.

## 3. Create New Thread After Request (`-t`)

From the slides:

```c
void createThreadForEachRequest(int serverSocket) {
    while (1) {
        int clientSocket = accept(serverSocket, &sockInfo, &alen);
        if (clientSocket >= 0) {
            pthread_attr_t attr;
            pthread_attr_init(&attr);
            pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_DETACHED);
            pthread_create(&thread, &attr,
                           dispatchHTTP, (void *) clientSocket);
        }
    }
}
```

- A new **detached thread** is created for each incoming connection.
- `PTHREAD_CREATE_DETACHED`: thread resources are recycled automatically when the thread ends — no `pthread_join()` needed.
- The thread goes away when the request is completed.

## 4. Pool of Threads (`-p`)

From the slides:

```c
void poolOfThreads(int serverSocket) {
    for (int i = 0; i < 4; i++) {
        pthread_create(&thread[i], NULL, loopthread, serverSocket);
    }
    loopthread(serverSocket);   // main thread also serves
}

void *loopthread(int serverSocket) {
    while (1) {
        int clientSocket = accept(serverSocket, &sockInfo, &alen);
        if (clientSocket >= 0) {
            dispatchHTTP(clientSocket);
        }
    }
}
```

- A **fixed pool** of threads (e.g., 5) is created at startup; each thread loops forever calling `accept()`.
- No thread creation overhead per request — threads are reused.

### The Thundering Herd Problem

From the slides: in Pool of Threads and Pool of Processes, sometimes the OS **does not allow** multiple threads/processes to call `accept()` on the same `masterSocket`. In other cases it allows it but with overhead: all waiting threads/processes wake up when one connection arrives, but only one can actually accept it.

**Fix**: wrap `accept()` in a mutex so only one thread waits on it at a time:

```c
mutex_lock(&mutex);
int clientSocket = accept(serverSocket, &sockInfo, 0);
mutex_unlock(&mutex);
```

From the Lab 5 handout: the pool size is **5 worker threads** in the lab implementation.

## 5. Pool of Processes

From the slides:

```c
void poolOfProcesses(int serverSocket) {
    for (int i = 0; i < 4; i++) {
        int pid = fork();
        if (pid == 0) {
            loopthread(serverSocket);
        }
    }
    loopthread(serverSocket);
}
```

- Same structure as Pool of Threads but uses forked **processes** instead.
- In the pool of processes, the **mutex must be created in shared memory** so it is visible across process boundaries.

## Command-Line Flags and Port Rules

From the slides (`myhttpd [-f|-t|-p] [<port>]`):

- No flag → iterative (single-threaded) mode
- `-f` → fork a new process per request
- `-t` → create a new thread per request
- `-p` → pool of threads
- Port number must be in range **1024 – 65535** (2¹⁶ − 1)
- Ports below 1024 require **root privilege**
- Compile with `-lpthread` when using pthreads

## Comparison

| Mode | Concurrency | Overhead | Notes |
|---|---|---|---|
| Iterative | None | None | Bottleneck: one at a time |
| Fork per request | Process per conn | High (fork) | Must reap zombies |
| Thread per request | Thread per conn | Medium (pthread_create) | DETACHED; auto-cleanup |
| Pool of threads | Fixed thread count | Low | Mutex around accept() |
| Pool of processes | Fixed process count | Low | Mutex in shared memory |

## Key Terms

| Term | Meaning |
|---|---|
| Iterative server | Handles one request at a time; simple but not concurrent |
| Fork server | Forks a child for each connection; parent immediately loops back |
| Thread-per-request | Creates a detached pthread for each connection |
| Thread pool | Fixed set of threads all calling accept() in a loop |
| `PTHREAD_CREATE_DETACHED` | Thread resources auto-freed on exit; no join needed |
| Thundering herd | All pooled threads/processes wake up for one connection |
| Mutex around accept() | Prevents thundering herd; only one thread blocks on accept() |
