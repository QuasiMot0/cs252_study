## The Problem NAT Solves

From the slides: IPv4 addresses are 32 bits, giving only ~4 billion addresses — not enough for every device. **NAT (Network Address Translation)** allows **multiple private hosts** to share a **single public IP address**.

## The 4-Tuple and NAT

From the slides: a TCP connection is uniquely identified by a **4-tuple**:

```
<src-ip, src-port, dst-ip, dst-port>
```

The NAT box uses this 4-tuple to track connections and translate addresses.

## How NAT Works — Outbound

From the slides, when a private host sends a packet to the internet:

1. Private host sends packet with source `<private-IP, private-port>`.
2. NAT box **rewrites** the source to `<IPnat, PORTrand>` where `IPnat` is the NAT box's public IP and `PORTrand` is a randomly chosen port.
3. NAT box **records the mapping** `<IPnat, PORTrand> ↔ <private-IP, private-port>` in the NAT table.
4. Packet goes out to the internet with the public source address.

## How NAT Works — Inbound (Return Traffic)

From the slides, when the reply arrives:

1. Reply arrives at `<IPnat, PORTrand>`.
2. NAT box **looks up** `<IPnat, PORTrand>` in the NAT table.
3. NAT box **reverse-translates** the destination back to `<private-IP, private-port>`.
4. Packet delivered to the private host.

## NAT as a Firewall

From the slides:

> "NAT box drops packets with no NAT table entry."

A packet arriving at the NAT box from the internet that does **not** match any entry in the NAT table is **dropped**. This means:

- Private hosts can initiate connections outbound (entries are created on outbound).
- External hosts **cannot** initiate inbound connections — there is no NAT table entry for them.

This makes NAT act as a natural firewall for inbound traffic.

## NAT Problems with Firewalls

From the slides: NAT-as-firewall creates problems for:

- **Peer-to-peer applications**: both peers may be behind NAT, so neither can initiate inbound connections to the other.
- **Server applications** behind NAT: external clients cannot reach a server that has a private IP.

## Port Forwarding (Static NAT Entries)

From the slides: **port forwarding** = creating a **static entry** in the NAT table so that inbound traffic on a specific public port is always forwarded to a specific private host and port.

Example: forward all traffic arriving at `<IPnat, port 80>` to `<192.168.1.5, port 80>` — this lets an internal web server be reachable from the internet.

## Example Network

From the slides:

- Private network: `192.168.1.0`
- NAT box public IP: `128.10.3.24`
- All traffic from the private network appears to come from `128.10.3.24` with various source ports

## NAT Configuration

From the slides: NAT is typically configured through a **wireless router's web interface**, which exposes options for:

- Viewing the current NAT table
- Adding port-forwarding (static) rules
- Enabling/disabling the firewall

## Key Terms

| Term | Meaning |
|---|---|
| NAT | Network Address Translation; allows multiple private hosts to share one public IP |
| 4-tuple | `<src-ip, src-port, dst-ip, dst-port>`; uniquely identifies a TCP connection |
| NAT table | Records mappings between private <IP:port> and public <IPnat:PORTrand> |
| IPnat | The NAT box's public IP address |
| PORTrand | Randomly chosen public port assigned to an outbound connection |
| Port forwarding | Static NAT entry; directs inbound traffic on a fixed public port to a private host |
| Private network | IP range not routable on the internet; e.g., 192.168.x.x |
| Thundering herd | Multiple threads/processes all waking up when one connection arrives |
