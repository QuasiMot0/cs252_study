## UDP — User Datagram Protocol

From the slides:

- **Unreliable**: packets may be lost, duplicated, or delivered out of order
- **Connectionless**: no setup before sending; no teardown after
- **Minimal overhead**: small header (source port, destination port, length, checksum)
- **Message-oriented**: each send is one distinct datagram
- **MTU**: maximum transmission unit is 1500 bytes per UDP packet
- Ports in header identify which application receives the datagram
- Good for **LAN communication** or **real-time** applications (e.g., video streaming) where low latency matters more than reliability

## TCP — Transmission Control Protocol

From the slides:

- **Reliable**: every byte is acknowledged; lost bytes are retransmitted
- **Connection-oriented**: connection established before data transfer (3-way handshake); torn down afterwards
- **Full-duplex**: data flows in both directions simultaneously
- **Stream interface**: data is a continuous byte stream with no message boundaries

## TCP's 6 Reliability Features

From the slides:

### 1. Adaptive Retransmission
TCP measures the round-trip time (RTT) for each acknowledgment and computes a retransmit timeout:

```
timeout = RTT + 4 × RTTSTDDEV
```

If a segment is not acknowledged within the timeout, it is retransmitted. The timeout adapts as network conditions change.

### 2. Cumulative Acknowledgments
An ACK for byte N confirms all bytes up to N have been received. This lets the sender retransmit only from the first unacknowledged byte.

### 3. Fast Retransmission
If the receiver gets an out-of-order segment, it sends a **duplicate ACK** for the last in-order byte. When the sender receives 3 duplicate ACKs, it retransmits the missing segment immediately — without waiting for the timeout.

### 4. Flow Control (Window)
The receiver advertises a **window size**: how many bytes it can buffer. The sender never has more than `window` bytes in flight. This prevents the sender from overwhelming the receiver.

### 5. Congestion Control
TCP avoids overwhelming the network with two phases:

- **Slow Start**: begin with a small congestion window; double it each RTT until a threshold.
- **Congestion Avoidance**: after the threshold, grow the window by 1 segment per RTT (linear growth).

On packet loss, the threshold is halved and slow start (or congestion avoidance) restarts.

### 6. Three-Way Handshake (Connection Setup)

From the slides:

| Step | Sender | Packet |
|---|---|---|
| 1 | Client → Server | SYN |
| 2 | Server → Client | SYN + ACK |
| 3 | Client → Server | ACK |

After step 3, the connection is established and data can flow.

## When to Use UDP vs TCP

From the slides:

| Scenario | Protocol |
|---|---|
| LAN communication, very low latency needed | UDP |
| Real-time audio/video streaming | UDP |
| Reliable file transfer, web pages, email | TCP |
| Any application requiring guaranteed delivery | TCP |

From the Lab 5 slides:
- **HTTP/1.1 and HTTP/2** run exclusively on **TCP**
- **HTTP/3** uses **QUIC**, a transport-layer protocol built on top of **UDP**, to improve communication speed and avoid head-of-line blocking

## Key Terms

| Term | Meaning |
|---|---|
| UDP | Unreliable, connectionless, message-oriented transport |
| TCP | Reliable, connection-oriented, stream-oriented transport |
| MTU | Maximum transmission unit; 1500 bytes for UDP |
| Adaptive retransmission | Timeout = RTT + 4×RTTSTDDEV; adjusts to network delay |
| Cumulative ACK | ACK N confirms all bytes up to N received |
| Fast retransmission | 3 duplicate ACKs → retransmit without waiting for timeout |
| Flow control | Receiver window limits bytes in flight |
| Slow Start | Begin with small congestion window; double each RTT until threshold |
| Congestion Avoidance | Linear window growth after threshold |
| Three-way handshake | SYN → SYN+ACK → ACK; establishes TCP connection |
