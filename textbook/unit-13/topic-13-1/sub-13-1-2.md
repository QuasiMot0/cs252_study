# Extreme Programming (XP)

## What is XP?

**Extreme Programming (XP)** is an agile software development methodology created by **Kent Beck** in the late 1990s. It takes proven software development practices and turns them up to the extreme (hence the name).

Key traits:
- **Iterative** development in short cycles (1-2 weeks)
- **Small teams** of 2-12 people
- Heavy emphasis on **automated testing**, **refactoring**, and **customer collaboration**

## The Four Activities

XP organizes work into four overlapping activities:

### 1. Planning
- Customer and developers play the **planning game**
- Customer writes **user stories** (small feature descriptions)
- Developers estimate effort for each story
- Together they produce:
  - A **release plan** (what features, in which release)
  - An **iteration plan** (what to build in the next 1-2 week cycle)

### 2. Designing
- Always use the **simplest design** that works now
- Don't anticipate future features you don't need yet (YAGNI)
- Refactor frequently as you learn more
- Use a **system metaphor** — a shared story describing how the system works

### 3. Coding
- **Pair programming**: two developers at one machine
  - Driver types, navigator reviews
  - Switch roles frequently
  - Improves quality, spreads knowledge
- Write only the code needed to make the current test pass
- **Continuous integration**: integrate multiple times per day; never let the build stay broken

### 4. Testing
- **Test-first development** (TDD): write the test before the code
- Tests define what "done" means
- Automated tests run continuously
- All tests must pass before integrating

## On-Site Customer

XP requires a real **on-site customer** — a product owner or user representative available daily to:
- Answer questions immediately
- Prioritize user stories
- Clarify requirements
- Accept or reject completed stories

This eliminates the multi-week feedback loop of traditional requirements documents.

## XP Rules Summary

| Activity | Rule |
|----------|------|
| Planning | Customer writes stories; developers estimate; prioritize together |
| Designing | Simplest thing that works; refactor when needed |
| Coding | Pair programming; integrate many times per day |
| Testing | Write tests first; all tests must pass before integration |

## XP vs Waterfall

| Aspect | Waterfall | XP |
|--------|-----------|-----|
| Requirements | Fully defined upfront | Evolve each iteration |
| Design | Complete before coding | Emergent, refactored |
| Testing | At the end | Continuous, test-first |
| Customer | At start and end | On-site every day |
| Delivery | Single big release | Frequent small releases |

## Key Takeaways

- XP = iterative, small teams (2-12), customer on-site
- Four activities: **Planning, Coding, Designing, Testing**
- **Pair programming**: two people, one keyboard
- **Test-first**: write the test before writing the code
- **Continuous integration**: merge and test multiple times per day
- **Simple design**: build what's needed now; refactor as you learn
- **Planning game**: produces release plan + iteration plan
