# Joel's Test: 12-Point Checklist for Software Teams

## What is Joel's Test?

Joel's Test is a 12-question yes/no checklist created by Joel Spolsky (co-founder of Stack Overflow, Trello) to quickly evaluate how good a software team's practices are. Each question is worth one point. Score interpretation:

- **12** — Perfect
- **11** — Tolerable
- **10 or below** — Serious problems

Most teams score 2–3. Many companies with good reputations score surprisingly low.

## The 12 Questions

1. **Do you use source control?**
   Without version control, you can't track changes, see history, or roll back mistakes.

2. **Can you make a build in one step?**
   The entire build process (compile, link, package) must be automatable with a single command. If it takes 12 manual steps, things will go wrong.

3. **Do you make daily builds?**
   Run automated builds every day to catch integration problems early, before they pile up. If the build breaks, fix it immediately.

4. **Do you have a bug database?**
   Every known bug must be tracked in a system (Bugzilla, Jira, GitHub Issues). Bugs kept in someone's head or in email will be forgotten.

5. **Do you fix bugs before writing new code?**
   Don't add features while bugs accumulate. A rising bug count signals poor code quality and creates technical debt that slows future work.

6. **Do you have an up-to-date schedule?**
   A schedule forces you to decide which features make the current release and which don't. Without one, you don't know if you're on track.

7. **Do you have a spec?**
   A specification written before coding prevents wasted work. Changing code is expensive; changing a spec is cheap.

8. **Do programmers have quiet working conditions?**
   Programming requires sustained concentration. Interruptions cost 15+ minutes of focus time each. Private offices or noise-canceling environments are essential.

9. **Do you use the best tools money can buy?**
   A programmer's time is expensive. Cheap hardware, slow machines, or outdated tools cost more in lost productivity than they save.

10. **Do you have testers?**
    If programmers spend time doing QA, they're not writing code. Dedicated testers find bugs more efficiently and keep programmers focused.

11. **Do new candidates write code during their interview?**
    You cannot evaluate a programmer without seeing them program. Ask them to write actual code — on a whiteboard, on a computer, anywhere.

12. **Do you do hallway usability testing?**
    Grab the next random person walking by and have them try to use your software. Watch without helping. Five such tests reveal most usability problems for almost zero cost.

## The Hallway Usability Test in Detail

The hallway usability test is informal and cheap:
- Find someone who doesn't know your product
- Ask them to complete a specific task using your software
- **Don't help them** — just watch where they get confused
- Five people reveal about 85% of usability problems

This is NOT an expensive formal usability lab study. You don't need cameras, observers, or a script. You can do it in the hallway in 15 minutes.

## Why This Matters

Joel's Test is intentionally simple. The point is that these 12 things are not hard to do — any team that cares about quality can do all 12. A low score is a red flag that the team doesn't value basic software engineering practices, which predicts poor product quality, high defect rates, and slow delivery.
