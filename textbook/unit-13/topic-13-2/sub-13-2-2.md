# Naming Code Smells

## Type Embedded in Name

**Problem:** The variable's type appears redundantly in its name.

```c
int intCount = 0;
char *strName = "Alice";
bool boolIsValid = true;
```

**Why it's bad:**
- The type is already visible in the declaration — saying it again adds no information
- If you change the type (e.g., `int` → `long`), you must also rename the variable
- It's a holdover from Hungarian notation, which was designed for weakly-typed languages

**Fix:**
```c
int count = 0;
char *name = "Alice";
bool isValid = true;
```

## Uncommunicative Name

**Problem:** Names that don't convey what the variable holds or what the method does.

```c
int x;
char *temp;
void doStuff(int a, int b);
```

**Why it's bad:** Readers must read the whole function to figure out what `x` is. This slows comprehension and increases bugs.

**Fix:** Name things for what they represent:

```c
int customerAge;
char *sortedResults;
void calculateDiscount(int price, int loyaltyYears);
```

The name should answer: "Why does this exist? What does it hold?"

## Inconsistent Names

**Problem:** The same concept is called by different names in different parts of the codebase.

```c
// Module A:
User *getUser(int id);

// Module B:
Customer *fetchCustomer(int id);

// Module C:
Account *loadUserRecord(int id);
```

All three do the same thing but use different terminology.

**Why it's bad:** Developers reading the code must constantly remember "oh, Customer and User are the same thing here." This cognitive load causes bugs and slows understanding.

**Fix:** Pick one name per concept and use it everywhere:

```c
User *getUser(int id);  // used in all three modules
```

## Dead Code

**Problem:** Code that is never executed.

Common examples:
```c
// Code after unconditional return:
int f() {
    return 42;
    printf("never reached\n");  // dead code
}

// Function that is never called:
void oldHelper() { ... }

// Branch that is always false:
if (0) {
    doSomething();  // dead code
}
```

**Why it's bad:** Dead code confuses readers ("is this important? Why is it here?"), requires maintenance effort, and can mislead debuggers.

**Fix:** Delete it. Git history preserves it if you ever need it again. **Do not comment it out** — commented-out code is just harder-to-spot dead code.

## Speculative Generality

**Problem:** Code written for hypothetical future requirements that don't exist yet.

```c
// "We might need to support plugins someday"
typedef void (*PluginHook)(void *);
PluginHook plugins[MAX_PLUGINS];  // unused

// "Maybe we'll need multiple strategies later"
typedef struct Strategy { ... } Strategy;
void setStrategy(Strategy *s) { ... }  // called with exactly one strategy
```

**Why it's bad:**
- Adds complexity now for zero current value
- Often turns out to be the wrong abstraction when the real requirement arrives
- Violates **YAGNI** (You Aren't Gonna Need It)

**Fix:** Delete it. When the actual requirement arrives, implement it with knowledge of what's really needed.

## Summary Table

| Smell | Example | Fix |
|-------|---------|-----|
| Type Embedded in Name | `intCount`, `strName` | Remove type from name: `count`, `name` |
| Uncommunicative Name | `x`, `temp`, `doStuff()` | Use descriptive names: `customerAge` |
| Inconsistent Names | `User` vs `Customer` for same concept | Pick one and use it everywhere |
| Dead Code | Code after return, unused functions | Delete it (git has history) |
| Speculative Generality | Abstractions for unused future features | YAGNI — delete it |

## Key Principle

Good names are the most powerful documentation tool you have. A codebase with clear, consistent names is dramatically easier to maintain than one with clever but opaque names.
