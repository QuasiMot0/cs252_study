# Structural Code Smells

## What is Refactoring?

**Refactoring** means changing the internal structure of code without changing its external behavior. Tests must still pass. The goal is to make code easier to understand, modify, and extend.

**Code smells** (Martin Fowler, Kent Beck) are surface indicators that something might be wrong with the code's design. They are not bugs — the code may work correctly — but they make the codebase harder to work with over time.

## Comments (as a Smell)

**Problem:** You need a comment to explain what the code does.

```c
// calculate the net salary after tax deduction
x = y * 0.85;
```

If the code were clear, no comment would be needed. The comment is a symptom of unclear code.

**Fix:** Rename and restructure so the code speaks for itself:

```c
net_salary = gross_salary * (1 - TAX_RATE);
```

**Note:** Comments explaining *why* (a workaround, a constraint, a subtle invariant) are valuable. Comments explaining *what* the code does are often a cover-up for unclear code.

## Long Method

**Problem:** A method has grown too large — hundreds of lines, doing many different things.

**Why it's bad:** Hard to read, hard to test, hard to reuse.

**Fix — Extract Method:** Pull logical sections out into smaller, well-named methods:

```c
// Before: 200-line processOrder() does everything
void processOrder(Order *o) {
    // validate inputs ...
    // calculate totals ...
    // apply discounts ...
    // update inventory ...
    // send confirmation email ...
}

// After: each concern is its own method
void processOrder(Order *o) {
    validateOrder(o);
    calculateTotals(o);
    applyDiscounts(o);
    updateInventory(o);
    sendConfirmation(o);
}
```

## Long Parameter List

**Problem:** A function takes too many parameters.

```c
void createUser(char *name, int age, char *email,
                char *address, char *phone, char *role);
```

**Why it's bad:** Hard to call correctly (easy to swap arguments), hard to remember, and long parameter lists often mean the function is doing too much.

**Fix:** Group related parameters into a struct or object:

```c
typedef struct {
    char *name;
    int age;
    char *email;
    char *address;
    char *phone;
    char *role;
} UserInfo;

void createUser(UserInfo *info);
```

## Duplicated Code

**Problem:** The same code structure appears in more than one place.

**Why it's bad:** Every bug fix or improvement must be made in every copy. Copies drift apart and become slightly different.

**Fix — Extract Method:** Pull the shared logic into a single function and call it from both places.

**Common locations:**
- Two methods in the same class doing the same thing
- Two subclasses with the same method body
- Two unrelated places in the codebase with identical logic

## Large Class

**Problem:** A class has grown enormous, with dozens of methods and hundreds of instance variables.

**Why it's bad:** It's doing too many things (violates Single Responsibility Principle), is hard to understand, and changes to one part risk breaking unrelated parts.

**Fix — Extract Class:** Move a coherent group of methods and fields into a new, smaller, focused class.

## Key Takeaways

| Smell | Problem | Fix |
|-------|---------|-----|
| Comments | Code needs explanation to be understood | Rename variables/methods; restructure |
| Long Method | Method does too many things | Extract Method — break into smaller methods |
| Long Parameter List | Too many parameters; hard to call | Introduce Parameter Object |
| Duplicated Code | Same logic in multiple places | Extract to shared method/base class |
| Large Class | Class has too many responsibilities | Extract Class |

**Refactoring** = improving code structure without changing behavior. Run tests before and after to verify.
