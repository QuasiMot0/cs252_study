# Proxy Pattern

## Intent

Provide a **surrogate or placeholder** for another object to control access to it.

**Category:** Structural

## The Problem

Sometimes you want to control access to an object. Maybe:
- The real object is expensive to create — don't create it until needed
- The real object is on a remote machine — you need local representation
- You want to check permissions before allowing access

Instead of the client accessing the real object directly, it goes through a **proxy** that intercepts calls and adds the needed logic.

## Structure

```
Client ──→ Subject (interface)
               ↑              ↑
          Proxy              RealSubject
       (wraps real)        (actual work)
```

Key insight: **Proxy and RealSubject both implement the same interface (Subject)**. The client doesn't know which one it has — it just calls the interface.

## Types of Proxies

### Virtual (Lazy) Proxy
Creates an expensive object only when it is first used.

**Example:** A document editor shows a page with 50 images. Loading all 50 at startup would be slow. Instead, each image slot holds a `ImageProxy`. The proxy creates the real `Image` object only when the user scrolls to that page.

```
// Client calls:
image.display();

// Proxy's display():
if (realImage == NULL) {
    realImage = loadFromDisk(filename);  // expensive — only once
}
realImage.display();
```

### Remote Proxy
Represents an object in a different address space (another process or machine).

The proxy handles all network communication. The client thinks it's calling a local object.

**Example:** RPC (Remote Procedure Call) stubs. In Java RMI, the stub on the client side is a remote proxy for the object on the server.

### Access-Control (Protection) Proxy
Checks permissions before forwarding calls to the real object.

```
void deleteFile(File *f) {
    if (!currentUser.hasPermission(DELETE)) {
        throw PermissionDenied;
    }
    realFileSystem.deleteFile(f);
}
```

### Logging / Caching Proxy
Logs calls or caches results:

```
int getWeather(Location loc) {
    if (cache.has(loc)) return cache.get(loc);
    int result = realWeatherService.getWeather(loc);
    cache.store(loc, result);
    return result;
}
```

## Proxy vs Decorator

Both patterns wrap an object and implement the same interface. The difference is **intent**:

| Aspect | Proxy | Decorator |
|--------|-------|-----------|
| Intent | Control *access* to the object | Add *behavior* to the object |
| Relationship to real object | Proxy manages the real object's lifecycle | Decorator receives the real object from outside |
| Typical use | Lazy init, access control, remoting | Adding features (logging, styling, formatting) |

## Example: Lazy Image Loading

```c
// Interface
typedef struct {
    void (*display)(void *self);
} Image;

// Real implementation
typedef struct {
    Image base;
    char *filename;
    // actual pixel data...
} RealImage;

void realImage_display(void *self) {
    RealImage *img = (RealImage *)self;
    // render pixels...
}

// Proxy
typedef struct {
    Image base;
    char *filename;
    RealImage *real;  // NULL until first use
} ImageProxy;

void imageProxy_display(void *self) {
    ImageProxy *proxy = (ImageProxy *)self;
    if (proxy->real == NULL) {
        proxy->real = loadFromDisk(proxy->filename);
    }
    proxy->real->base.display(proxy->real);
}
```

The client calls `display()` on an `Image *` — it doesn't know if it's a proxy or a real image.

## Key Takeaways

- Proxy provides a **surrogate** that controls access to a real object
- Proxy implements the **same interface** as the real object → client is unaware
- Three main types: **virtual** (lazy init), **remote** (network), **access-control** (permissions)
- Pattern is **Structural** (GoF category)
- Proxy vs Decorator: Proxy = access control; Decorator = behavior addition
