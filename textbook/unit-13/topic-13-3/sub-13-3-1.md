# Design Patterns Overview

## What is a Design Pattern?

A **design pattern** is a named, reusable solution to a commonly occurring software design problem. It is:

- **Not code** you copy and paste
- A **template** or **blueprint** you adapt to your specific situation
- A **shared vocabulary** that lets developers communicate design decisions efficiently

When you say "I used the Proxy pattern here," an experienced developer immediately understands the structure, intent, and trade-offs — without reading a paragraph of explanation.

## The Gang of Four (GoF95)

The foundational book on design patterns is:

> **"Design Patterns: Elements of Reusable Object-Oriented Software"** (1995)  
> Gamma, Helm, Johnson, Vlissides — the "Gang of Four"

It catalogued **23 fundamental OO design patterns**, organized into three categories. This book, often just called "the GoF book," is one of the most influential books in software engineering.

## The Three Categories

### Creational Patterns
Deal with **object creation** — how objects are instantiated.

| Pattern | Purpose |
|---------|---------|
| Singleton | Ensure only one instance exists |
| Factory Method | Let subclasses decide which class to instantiate |
| Builder | Construct complex objects step by step |
| Prototype | Create objects by cloning an existing one |

### Structural Patterns
Deal with **how objects are composed** — assembling objects into larger structures.

| Pattern | Purpose |
|---------|---------|
| **Proxy** | Control access to an object via a surrogate |
| Adapter | Convert one interface to another |
| Decorator | Add behavior to objects without changing their class |
| Composite | Treat individual objects and compositions uniformly |
| Facade | Provide a simplified interface to a complex subsystem |

### Behavioral Patterns
Deal with **how objects communicate** — assigning responsibilities and defining communication protocols.

| Pattern | Purpose |
|---------|---------|
| **Command** | Encapsulate a request as an object |
| Observer | Notify dependents when an object changes state |
| Strategy | Define a family of algorithms; make them interchangeable |
| Iterator | Access elements of a collection sequentially |
| Template Method | Define a skeleton algorithm; subclasses fill in steps |

## Anatomy of a Pattern

Each GoF pattern is documented with:

- **Intent** — What problem does it solve? (one-sentence summary)
- **Applicability** — When should you use it?
- **Structure** — UML class diagram
- **Participants** — The classes/objects involved and their roles
- **Consequences** — Trade-offs: what you gain and what it costs
- **Implementation** — Hints and language-specific issues
- **Known Uses** — Real examples in existing systems

The **consequences** section is especially important: it tells you the trade-offs so you can decide if the pattern is appropriate for your situation.

## Are Patterns Language-Specific?

No. GoF patterns are **language-independent concepts** applicable to any object-oriented language (Java, C++, Python, C#, etc.). The *structure* of the pattern is the same; only implementation details differ by language.

Some languages have features that make certain patterns trivial (e.g., Python's first-class functions can replace many Command implementations), but the underlying concept still applies.

## When to Use Patterns

Patterns are tools, not rules. Don't apply a pattern just because you recognize a superficial similarity. Ask:

1. Do I have this problem?
2. Does this pattern's **intent** match my problem?
3. Are the **consequences** acceptable for my situation?

Over-applying patterns leads to over-engineered, hard-to-read code. Under-applying them leads to reinventing solutions to already-solved problems.

## Key Takeaways

- GoF95 = 23 patterns by Gamma, Helm, Johnson, Vlissides
- 3 categories: **Creational** (creation), **Structural** (composition), **Behavioral** (communication)
- Proxy = Structural; Command = Behavioral
- Patterns are **not code** — they are reusable solution templates
- Patterns provide a **shared vocabulary** for design communication
- Always check the **consequences** section before applying a pattern
