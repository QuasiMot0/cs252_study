# Command Pattern

## Intent

**Encapsulate a request as an object**, letting you parameterize clients with different requests, queue or log requests, and support undoable operations.

**Category:** Behavioral

## The Problem

Imagine a GUI application. A "Save" button, a "Save" menu item, and a Ctrl+S keyboard shortcut all do the same thing. How do you avoid duplicating the save logic in three places?

Or: How do you implement undo/redo, where any operation can be reversed?

The Command pattern solves both by wrapping each action in an object.

## Structure

```
Invoker ──→ Command (interface)
              execute()
              undo()
                 ↑
           ConcreteCommand
              execute() → calls Receiver
              undo()    → reverses action
                              ↓
                          Receiver
                       (does actual work)
```

## Participants

| Role | Responsibility |
|------|---------------|
| **Command** | Interface with `execute()` (and optionally `undo()`) |
| **ConcreteCommand** | Implements `execute()` by calling the Receiver; stores state needed for undo |
| **Receiver** | The object that knows how to do the actual work |
| **Invoker** | Holds and triggers commands (button, menu item, scheduler) |
| **Client** | Creates ConcreteCommand objects and wires them to Invokers |

## Benefits

### 1. Undo / Redo

Each ConcreteCommand stores the state needed to reverse itself:

```
History stack: [TypeCommand("Hello"), BoldCommand(), InsertImageCommand]

User presses Undo:
  InsertImageCommand.undo()  → removes image
  
User presses Undo again:
  BoldCommand.undo()         → removes bold
```

### 2. Queuing and Scheduling

Commands are objects — you can store them in a queue and execute later:

```
// Job scheduler
queue.add(new EmailCommand(recipient, body));
queue.add(new ResizeImageCommand(img, 800, 600));
// ... execute all at midnight
```

### 3. Logging / Crash Recovery

Serialize commands to disk as they execute. On crash, reload and replay `execute()` on each saved command to restore state. This is the basis of **write-ahead logging** and **event sourcing**.

### 4. Macro Commands

A command that contains a list of sub-commands:

```c
void macroCommand_execute(MacroCommand *m) {
    for (int i = 0; i < m->count; i++) {
        m->commands[i]->execute(m->commands[i]);
    }
}
```

This is the Composite pattern applied to commands.

## GUI Example

In a text editor, every menu item holds a Command object:

```
File > Save     → SaveCommand.execute() → editor.save()
File > Open     → OpenCommand.execute() → editor.open()
Edit > Copy     → CopyCommand.execute() → editor.copy()
Edit > Paste    → PasteCommand.execute() → editor.paste()
```

The menu code just calls `command.execute()`. To add a new menu item, create a new ConcreteCommand — no changes to the menu itself. To support Ctrl+S as an alternative to File > Save, bind the same `SaveCommand` to the keyboard handler.

## Decoupling

The Invoker (button/menu/scheduler) doesn't know what the Command does — it just calls `execute()`. The Receiver (TextEditor, FileSystem) doesn't know it's being driven by commands. This decoupling makes the system easy to extend.

## Example: Simple Command Pattern

```c
// Command interface
typedef struct Command {
    void (*execute)(struct Command *self);
    void (*undo)(struct Command *self);
} Command;

// Receiver
typedef struct {
    int value;
} Counter;

// ConcreteCommand
typedef struct {
    Command base;
    Counter *counter;
    int amount;  // stored for undo
} IncrementCommand;

void increment_execute(Command *self) {
    IncrementCommand *cmd = (IncrementCommand *)self;
    cmd->counter->value += cmd->amount;
}

void increment_undo(Command *self) {
    IncrementCommand *cmd = (IncrementCommand *)self;
    cmd->counter->value -= cmd->amount;
}
```

The Invoker calls `cmd->execute(cmd)` to do, and `cmd->undo(cmd)` to reverse.

## What Command Does NOT Do

The Command pattern does **not** eliminate the Receiver. The Receiver is the object that actually performs the work. Command just wraps the *call* to the Receiver as an object.

## Key Takeaways

- Command encapsulates a **request as an object** with `execute()` (and `undo()`)
- **Invoker** holds commands and calls `execute()` — decoupled from what the command does
- **Receiver** does the actual work — called by the command's `execute()`
- Enables: **undo/redo** (history stack), **queuing**, **logging**, **macro commands**
- Category: **Behavioral** (GoF)
- Each menu item/button holding a Command object is the classic application
