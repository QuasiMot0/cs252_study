## The Lock Ordering Rule

From the slides:
> "Assign an order to mutexes. If i < j, always lock mutex mi before mj."

If all threads always acquire mutexes in the same global order (lowest index first), **deadlock is impossible**.

## Proof by Contradiction

From the slides:

Assume a deadlock exists. Then there is a cycle in the resource graph. Follow the cycle:

- Some thread T_k is waiting for a mutex m_j held by thread T_{k+1}.
- T_{k+1} holds m_j and is waiting for some m_i held by the next thread in the cycle.
- For there to be a cycle, at some point a thread holds a higher-indexed mutex and is waiting for a lower-indexed mutex.
- But the rule says **always lock lower-indexed mutex first** — you cannot hold m_j and then try to lock m_i if i < j.
- Contradiction. Therefore no cycle exists, and deadlock is impossible.

## Fixing the Bank Account Example

From the slides, the deadlock occurred because `Transfer1_to_2` locked m1 then m2, while `Transfer2_to_1` locked m2 then m1 (opposite order).

**Fix**: both functions always lock m1 before m2:

```c
Transfer1_to_2() {
    mutex_lock(&m1);    // i=1 < j=2: lock m1 first
    mutex_lock(&m2);
    // transfer
    mutex_unlock(&m2);
    mutex_unlock(&m1);
}

Transfer2_to_1() {
    mutex_lock(&m1);    // i=1 < j=2: lock m1 first (even though transferring from 2 to 1)
    mutex_lock(&m2);
    // transfer
    mutex_unlock(&m2);
    mutex_unlock(&m1);
}
```

## Generic Transfer Function

From the slides:

```c
Transfer_i_to_j(int i, int j) {
    if (i < j) {
        mutex_lock(&m[i]);
        mutex_lock(&m[j]);
    } else {
        mutex_lock(&m[j]);
        mutex_lock(&m[i]);
    }
    // transfer funds
    mutex_unlock(&m[i]);
    mutex_unlock(&m[j]);
}
```

Always lock the lower-indexed account mutex first, regardless of the direction of the transfer.

## Unlock Ordering Does Not Matter

From the slides:
> "The order in which you unlock mutexes does not matter for deadlock prevention."

Deadlock requires a cycle of blocked threads. A thread that is unlocking is not blocked — it is running. So unlock order cannot create a deadlock.

## Dining Philosophers Problem

From the slides:

N philosophers sit at a round table. Each philosopher alternates between thinking and eating. To eat, a philosopher needs **both** the fork to their left and the fork to their right. There are N forks (one between each pair of adjacent philosophers).

**Deadlock scenario**:
1. Every philosopher picks up their **right** fork simultaneously.
2. Every philosopher then waits for their **left** fork (held by the neighbor).
3. No one can proceed — deadlock.

**Prevention using lock ordering**:

Assign a number to each fork (0 through N-1). Each philosopher must always pick up the **lower-numbered** fork first.

- All philosophers except the one between fork N-1 and fork 0 pick up right then left (or left then right, depending on which is lower-numbered).
- The philosopher between fork N-1 and fork 0 picks up fork 0 first (lower number) instead of fork N-1 — this breaks the cycle.

This ensures no cycle can form in the resource graph.

## Key Terms

| Term | Meaning |
|---|---|
| Lock ordering | Global rule: always acquire mutex m_i before m_j if i < j |
| Deadlock prevention | Technique that makes deadlock structurally impossible |
| Dining philosophers | Classic deadlock scenario: N threads each need 2 shared resources |
| Cycle prevention | Lock ordering prevents cycles in the resource graph; no cycle = no deadlock |
| Unlock ordering | Does not affect deadlock; only acquisition order matters |
