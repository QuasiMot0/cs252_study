## Deadlock vs. Starvation

From the slides:
> "Deadlock: when a thread blocks forever. You have to kill the process."
> "Starvation: when a thread waits for a long time. Less serious than deadlock."

| Condition | Description | Example |
|---|---|---|
| Deadlock | Thread blocks forever; process must be killed | Circular mutex wait |
| Starvation | Thread waits a very long time but may eventually proceed | Writer starvation in read-write locks |

## Example of a Deadlock: Bank Account Transfer

From the slides:

```c
// Thread 1: Transfer from account 1 to account 2
Transfer1_to_2() {
    mutex_lock(&m1);
    mutex_lock(&m2);
    // transfer funds
    mutex_unlock(&m2);
    mutex_unlock(&m1);
}

// Thread 2: Transfer from account 2 to account 1
Transfer2_to_1() {
    mutex_lock(&m2);
    mutex_lock(&m1);
    // transfer funds
    mutex_unlock(&m1);
    mutex_unlock(&m2);
}
```

Deadlock scenario:

| Step | Thread 1 | Thread 2 | m1 | m2 |
|---|---|---|---|---|
| 1 | `mutex_lock(&m1)` — acquired | — | T1 | free |
| 2 | — | `mutex_lock(&m2)` — acquired | T1 | T2 |
| 3 | `mutex_lock(&m2)` — **blocks** (held by T2) | — | T1 | T2 |
| 4 | blocked | `mutex_lock(&m1)` — **blocks** (held by T1) | T1 | T2 |

Thread 1 holds m1, waits for m2. Thread 2 holds m2, waits for m1. Neither can proceed — **deadlock**.

## Graph Representation of Deadlock

From the slides:

- **T → M**: Thread T is **waiting** for mutex M
- **M → T**: Mutex M is **held by** thread T
- **Deadlock = a cycle in this graph**

For the bank example:

```
m1 → T1 → m2 → T2 → m1
```

This cycle means T1 holds m1 and waits for m2, while T2 holds m2 and waits for m1.

## Larger Deadlock Example

From the slides, a deadlock can involve more threads and mutexes:

```
T1 → m2 → T2 → m3 → T3 → m4 → T4 → m1 → T1
```

Four threads, four mutexes, one cycle — all four threads block forever.

## Debugging Deadlock with gdb

From the slides: when a program hangs due to suspected deadlock:

```bash
gdb ./program <pid>
info threads        # list all threads
thread <n>          # switch to thread n
bt                  # backtrace — shows where each thread is blocked
```

By inspecting the backtrace of each thread, you can identify which mutexes each thread is waiting on and which it holds, allowing you to trace the cycle manually.

## Key Terms

| Term | Meaning |
|---|---|
| Deadlock | State where threads block forever waiting for each other; process must be killed |
| Starvation | State where a thread waits a long time but may eventually proceed |
| Resource graph | Graph where T→M means T waits for M and M→T means M held by T |
| Cycle in resource graph | Necessary and sufficient condition for deadlock |
| Circular wait | Each thread in a cycle holds a resource that the next thread needs |
