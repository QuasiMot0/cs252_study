## What Are Condition Variables?

From the slides:
> "Condition Variables is a mechanism used for synchronization like semaphores."
> "Condition variables and semaphores are equivalent in the sense that they can solve the same problems of synchronization."
> "Condition variables are more expressive than semaphores but using one vs. the other is a matter of taste."

Condition variables provide a way for a thread to **wait until a condition becomes true**, while automatically releasing and re-acquiring a lock.

## Atomic Sections in Java

From the slides: in Java, atomic sections are implemented using the `synchronized` keyword. No explicit calls to `mutex_lock()`/`mutex_unlock()` are needed. **Any Java object can serve as a lock.**

Java implements synchronization in three ways:

### Synchronized Class Method

```java
class class_name {
    static synchronized type method_name1() {
        statement block
    }
    static synchronized type method_name2() {
        statement block
    }
}
```

All the statements in the method become the atomic section, and the **Class object** is the lock. All static synchronized methods in the class are mutually exclusive.

### Synchronized Instance Method

```java
class class_name {
    synchronized type method_name1() {
        statement block
    }
    synchronized type method_name2() {
        statement block
    }
}
```

All the statements in the method are part of the atomic section, and the **instance object (`this`)** is the lock. All synchronized methods in the instance are mutually exclusive.

### Synchronized Statement

```java
class class_name {
    type method_name() {
        synchronized (object) {
            statement block
        }
    }
}
```

All statements inside the defined synchronized block become an atomic section. The object in the parenthesis is the lock. All synchronized statements that share the same object will be mutually exclusive.

## Condition Variables in Java

From the slides: inside a synchronized block that uses an **object** as the lock, you can use:

| Method | Effect |
|---|---|
| `object.wait()` | Caller waits until another thread calls `notify` or `notifyAll` on the object; automatically releases the lock while waiting and re-acquires it on wake-up |
| `object.notify()` | Wakes up **one** thread waiting in `object.wait()` |
| `object.notifyAll()` | Wakes up **all** threads waiting in `object.wait()` |

## Bounded Buffer in Java

From the slides, a BoundedBuffer implemented with condition variables instead of semaphores:

```java
class BoundedBuffer {
    int[] buffer;
    int tail, head, n, max;
    final Object monitor = new Object();  // serves as mutex and condition variable

    public BoundedBuffer(int max) {
        buffer = new int[max];
        tail = 0; head = 0; n = 0;
        this.max = max;
    }

    public void enqueue(int v) {
        synchronized(monitor) {
            while (n == max) {           // wait if buffer is full
                try { monitor.wait(); }
                catch (InterruptedException e) {}
            }
            buffer[tail] = v;
            tail = (tail + 1) % max;
            n++;
            monitor.notifyAll();         // wake up any thread waiting in dequeue
        }
    }

    public int dequeue() {
        int v;
        synchronized(monitor) {
            while (n == 0) {             // wait if buffer is empty
                try { monitor.wait(); }
                catch (InterruptedException e) {}
            }
            v = buffer[head];
            head = (head + 1) % max;
            n--;
            monitor.notifyAll();         // wake up any thread waiting in enqueue/dequeue
        }
        return v;
    }
}
```

## The while Loop Pattern (Spurious Wakeups)

From the slides, both `enqueue` and `dequeue` use **`while`** (not `if`) to check the condition:

```java
while (n == max) { monitor.wait(); }
```

This is necessary because:
1. **Spurious wakeups**: a thread may be woken up even when the condition has not changed.
2. **Multiple waiters**: `notifyAll()` wakes all waiting threads, but only one of them may find the condition true. The rest must loop back and wait again.

Using `if` instead of `while` is a bug — the thread would proceed without verifying the condition is actually satisfied.

From the slides:

```

Consumer Thread Running
produce(0)

produce(9)

consume(0)

produce(26732)

```

The producer fills the buffer (max=10), then blocks on `monitor.wait()` inside `enqueue`. As the consumer removes items and calls `notifyAll()`, the producer unblocks, adds one item, and calls `notifyAll()` to wake the consumer.

## Key Terms

| Term | Meaning |
|---|---|
| Condition variable | Synchronization mechanism where a thread waits until a condition is signaled |
| `synchronized` | Java keyword that makes a method or block an atomic section using an object as the lock |
| `object.wait()` | Releases the lock and blocks until notified; re-acquires lock on wake-up |
| `object.notify()` | Wakes one thread waiting on `object.wait()` |
| `object.notifyAll()` | Wakes all threads waiting on `object.wait()` |
| Spurious wakeup | Wake-up that occurs without a corresponding notify; handled by using `while` instead of `if` |
| Monitor | Java object used as both a mutex and a condition variable |
