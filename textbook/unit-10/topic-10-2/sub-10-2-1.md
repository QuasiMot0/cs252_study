## The Problem: Shared State Without Coordination

From the slides:
> "What happens if two threads run counter++ at the same time?"

```c
int counter = 0;   // global variable

void* count(void* arg) {
    int i;
    for (i = 0; i < 100; i++) {
        counter++;
    }
}
```

Two threads each running this loop should produce `counter = 200`. In practice, the result is unpredictable.

## Why counter++ Is Not Atomic

From the slides:
> "counter++ is not an atomic operation."
> "counter++ is actually:"
```
int tmp = counter;
tmp = tmp + 1;
counter = tmp;
```

The compiler generates three separate machine instructions: **load**, **add**, **store**. A context switch can occur between any two of these instructions.

## The Lost Update Trace

From the slides, a concrete example of the race condition:

| Step | Thread 1 | Thread 2 | counter value |
|---|---|---|---|
| 1 | `tmp1 = counter` (tmp1 = 0) | — | 0 |
| 2 | *context switch* | — | 0 |
| 3 | — | runs 23 iterations | 23 |
| 4 | *context switch back* | — | 23 |
| 5 | `tmp1 = tmp1 + 1` (tmp1 = 1) | — | 23 |
| 6 | `counter = tmp1` | — | **1** |

23 increments are lost because Thread 1 loaded a stale value before Thread 2 ran, then overwrote Thread 2's result.

## Atomic Section / Critical Section

From the slides:
> "We need to make the update of counter atomic."
> "An atomic section is a section of code that only one thread should execute at a time."
> "This section of code is also called a critical section."

The three instructions (load / add / store) must be grouped so that no other thread can execute them simultaneously.

## Key Terms

| Term | Meaning |
|---|---|
| Race condition | Non-deterministic result when multiple threads access shared state concurrently |
| Atomic operation | An operation that completes without any possible interruption |
| Critical section | Code that must be executed by only one thread at a time |
| Lost update | Race condition where one thread's write overwrites another thread's concurrent update |
