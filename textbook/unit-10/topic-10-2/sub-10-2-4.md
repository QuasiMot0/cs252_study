## The Problem: Implementing a Lock

To implement a mutex lock, you need to check whether the lock is free and then acquire it as a single uninterruptible operation. Doing this in two separate steps (check, then set) would itself be a race condition.

## Two Approaches to Mutex Implementation

From the slides:

### Approach 1: Disable Interrupts

```c
mutex_lock() {
    disable_interrupts();
    // critical section
}
mutex_unlock() {
    enable_interrupts();
}
```

Problems from the slides:
- Only works in the **kernel** (user processes cannot disable interrupts)
- Does **not work on multi-CPU** systems — disabling interrupts on one CPU does not stop another CPU from accessing shared memory simultaneously
- Disabling interrupts is slow

### Approach 2: Spin Locks (test_and_set)

From the slides:

```c
int test_and_set(int *lock) {
    int old = *lock;
    *lock = 1;
    return old;
}
```

`test_and_set` is a **hardware instruction** that atomically reads the old value and writes `1` in a single CPU operation. No context switch can occur between the read and the write.

## Mutex Lock Built on a Spin Lock

From the slides, a mutex is implemented using a spin lock plus a wait queue:

```
mutex_lock(m):
    spinlock(&m->slock)
    if (m->locked):
        add self to m->queue
        spinunlock(&m->slock)
        thr_yield()        // go to sleep
    else:
        m->locked = true
        spinunlock(&m->slock)

mutex_unlock(m):
    spinlock(&m->slock)
    if (m->queue not empty):
        t = dequeue(m->queue)
        wake_up(t)         // next thread gets the lock
    else:
        m->locked = false
    spinunlock(&m->slock)
```

The spin lock (`m->slock`) is held only briefly — just long enough to inspect and update the mutex state, then released. The actual waiting (sleeping) happens outside the spin lock.

## Why Mutex Is Fair but Spin Lock Is Not

From the slides:
- **Mutex**: when unlocked, the thread that has been waiting the **longest** (head of the queue) is woken up — FIFO order.
- **Spin lock**: any thread calling `test_and_set` at the right moment acquires the lock — no guarantee of ordering.

## Key Terms

| Term | Meaning |
|---|---|
| `test_and_set` | Hardware atomic instruction: reads old value + sets to 1 in one step |
| Disable interrupts | Approach 1 for mutex; only works in kernel and fails on multi-CPU |
| Spin lock | Busy-wait lock using `test_and_set`; used internally by mutex |
| Wait queue | List of sleeping threads waiting for a mutex; enables FIFO (fair) wake-up |
| Atomic | An operation that cannot be interrupted mid-execution |
