## What Is a Spin Lock?

A spin lock is a locking mechanism where a thread **repeatedly tests** the lock until it becomes free (busy-waiting). Unlike a mutex, the thread does not go to sleep — it stays running and keeps checking.

## test_and_set

From the slides:

```c
int test_and_set(int *lock) {
    int old = *lock;
    *lock = 1;
    return old;
}
```

`test_and_set` reads the current value of `*lock` and atomically sets it to `1`. It returns the **old** value. If the old value was `0`, the caller acquired the lock. If the old value was `1`, the lock was already held.

This operation is implemented in **hardware** (a single CPU instruction), making it truly atomic — no context switch can occur between the read and the write.

## spinlock() and spinunlock()

From the slides:

```c
void spinlock(int *lock) {
    while (test_and_set(lock) == 1) {
        thr_yield();
    }
}

void spinunlock(int *lock) {
    *lock = 0;
}
```

- `spinlock`: loops calling `test_and_set` until it gets back `0` (lock was free and is now acquired).
- `thr_yield()`: voluntarily gives up the CPU so other threads can run, reducing wasted CPU cycles compared to a tight spin.
- `spinunlock`: sets the lock variable back to `0` (released).

## Example Trace

From the slides:

| Step | Thread 1 | Thread 2 | lock |
|---|---|---|---|
| 1 | `test_and_set` → old=0, sets lock=1, returns 0 → acquired | — | 1 |
| 2 | critical section | — | 1 |
| 3 | — | `test_and_set` → old=1, returns 1 → loops + yield | 1 |
| 4 | `spinunlock` → lock=0 | — | 0 |
| 5 | — | `test_and_set` → old=0, sets lock=1, returns 0 → acquired | 1 |

## Spin Lock vs. Mutex

From the slides:

| Property | Spin Lock | Mutex |
|---|---|---|
| Waiting thread behavior | Busy-waits (runs, calls thr_yield) | Sleeps (put in wait queue) |
| CPU usage while waiting | High (wastes CPU) | None |
| Best for | Short critical sections | Long critical sections |
| Fairness | **Not fair** — no guaranteed order | **Fair** — FIFO wake-up |
| Implementation | Directly uses `test_and_set` | Built on top of a spin lock + wait queue |

From the slides:
> "A spinlock is not fair. There is no guarantee that threads get the lock in the order they asked for it."

## When to Prefer Spin Locks

From the slides: spin locks are appropriate when:
- The critical section is **very short** (so the spin is brief)
- The thread is **unlikely to be preempted** while holding the lock
- Sleeping (mutex overhead) would cost more than spinning

Mutexes are preferred for longer critical sections because they put the waiting thread to sleep instead of wasting CPU cycles.

## Key Terms

| Term | Meaning |
|---|---|
| Spin lock | Lock where the waiting thread busy-waits instead of sleeping |
| `test_and_set` | Hardware atomic instruction: reads old value and sets to 1 in one step |
| `thr_yield()` | Voluntarily gives up the CPU; used inside spinlock to reduce wasted cycles |
| `spinunlock` | Sets lock variable to 0 (releases the lock) |
| Fairness | Whether threads acquire the lock in the order they requested it; spin locks are NOT fair |
