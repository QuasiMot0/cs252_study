## What Is a Semaphore?

From the slides:
> "A semaphore is a generalization of a mutex. It has a counter that is >= 0."

Unlike a mutex (which is simply locked or unlocked), a semaphore holds a **counter** that controls how many threads can proceed simultaneously.

## POSIX Semaphore API

```c
sema_init(&sem, n);    // initialize with count n
sema_wait(&sem);       // decrement; block if count < 0
sema_post(&sem);       // increment; wake up one waiting thread
```

| Function | Effect |
|---|---|
| `sema_init(&sem, n)` | Set the semaphore counter to `n` |
| `sema_wait(&sem)` | Decrement counter; if counter < 0, block the calling thread |
| `sema_post(&sem)` | Increment counter; if counter ≤ 0, wake up the longest-waiting thread |

## How sema_wait and sema_post Work

From the slides:

```
sema_wait(sem):
    sem->count--
    if (sem->count < 0):
        block this thread

sema_post(sem):
    sem->count++
    if (sem->count <= 0):
        wake up the longest-waiting thread
```

`sema_post` wakes the thread that has been waiting the **longest** (FIFO order).

## Three Use Cases Based on Initial Count

From the slides:

### n = 1 (Binary Semaphore / Mutex Equivalent)

```c
sema_init(&sem, 1);
sema_wait(&sem);   // like mutex_lock
// critical section
sema_post(&sem);   // like mutex_unlock
```

Only one thread can be in the critical section at a time. Equivalent to a mutex.

### n > 1 (N Resources)

```c
sema_init(&sem, 3);  // 3 printers available
sema_wait(&sem);     // acquire one printer
// use printer
sema_post(&sem);     // release printer
```

Up to `n` threads can proceed simultaneously. The 4th thread blocks until one of the first three calls `sema_post`.

### n = 0 (Wait for Event / Synchronization)

```c
sema_init(&sem, 0);

// Thread A (waiter):
sema_wait(&sem);    // blocks immediately (count goes to -1)
// proceeds after thread B posts

// Thread B (signaler):
// ... do work ...
sema_post(&sem);    // wake up thread A
```

Used for synchronization: one thread waits until another signals that an event has occurred.

## A Synchronized List Class

From the slides, a drawback of a plain mutex-protected list is that `remove()` returns NULL when the list is empty. With a semaphore, `remove()` can **block until an item is available**.

```c
class SyncList {
    List _list;
    Semaphore _emptySem;   // count = # items in list
    Mutex _mutex;
};

SyncList() {
    sema_init(&_emptySem, 0);
}

void insert(int val) {
    mutex_lock(&_mutex);
    _list.insert(val);
    mutex_unlock(&_mutex);
    sema_post(&_emptySem);     // signal: one more item
}

int remove() {
    sema_wait(&_emptySem);     // wait until item exists
    mutex_lock(&_mutex);
    int val = _list.remove();
    mutex_unlock(&_mutex);
    return val;
}
```

**Critical ordering rule**: `sema_wait` must be called **outside** (before) the mutex lock. If `sema_wait` is called inside the mutex, a deadlock results:
- `remove()` holds the mutex and calls `sema_wait`, blocking.
- `insert()` tries to acquire the mutex to add an item — but `remove()` already holds it.
- Neither can proceed.

## Key Terms

| Term | Meaning |
|---|---|
| Semaphore | Synchronization primitive with a counter; generalizes a mutex |
| `sema_wait` | Decrements counter; blocks if counter < 0 |
| `sema_post` | Increments counter; wakes longest-waiting thread if counter ≤ 0 |
| Binary semaphore | Semaphore with initial count 1; equivalent to a mutex |
| Counting semaphore | Semaphore with initial count n > 1; allows n simultaneous threads |
| Synchronization semaphore | Semaphore with initial count 0; used to wait for an event |
| `_emptySem` | Semaphore tracking the number of items in a list; init = 0 |
