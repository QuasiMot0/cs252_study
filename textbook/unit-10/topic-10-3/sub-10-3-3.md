## Motivation

From the slides:
> "Readers can run concurrently with other readers. But a writer needs exclusive access."

A plain mutex would force readers to be sequential even though concurrent reads are safe. A read-write lock allows:
- **Multiple readers** to hold the lock simultaneously
- **One writer** at a time, with exclusive access (no readers allowed)

## State Variables

From the slides:

```c
class RWLock {
    int _nreaders;           // number of active readers (init 0)
    Semaphore _semAccess;    // controls writer access (init 1)
    Mutex _mutex;            // protects _nreaders
};
```

## API and Implementation

### readLock()

```c
void readLock() {
    mutex_lock(&_mutex);
    _nreaders++;
    if (_nreaders == 1) {
        sema_wait(&_semAccess);   // first reader blocks writers
    }
    mutex_unlock(&_mutex);
}
```

The **first reader** calls `sema_wait(&_semAccess)`, which prevents any writer from entering. Subsequent readers increment `_nreaders` but do not need to touch `_semAccess`.

### readUnlock()

```c
void readUnlock() {
    mutex_lock(&_mutex);
    _nreaders--;
    if (_nreaders == 0) {
        sema_post(&_semAccess);   // last reader unblocks writers
    }
    mutex_unlock(&_mutex);
}
```

The **last reader** calls `sema_post(&_semAccess)`, which allows a waiting writer to proceed.

### writeLock()

```c
void writeLock() {
    sema_wait(&_semAccess);   // exclusive access: blocks all readers + writers
}
```

### writeUnlock()

```c
void writeUnlock() {
    sema_post(&_semAccess);
}
```

## Writer Starvation

From the slides:
> "Writer starvation: if there are always readers, the writer will never get access."

If new readers keep arriving and incrementing `_nreaders` before it reaches 0, the writer remains blocked on `sema_wait(&_semAccess)` indefinitely. This is a known limitation of this implementation.

From the slides: writer starvation is classified as **starvation** (waits a long time), not deadlock (blocks forever until killed), because it would eventually resolve if readers stopped.

## Key Terms

| Term | Meaning |
|---|---|
| Read-write lock | Lock allowing concurrent readers but exclusive writers |
| `_nreaders` | Count of active readers; first reader takes semaphore, last releases it |
| `_semAccess` | Semaphore (init=1) that controls exclusive writer access |
| Writer starvation | Writer never gets access because readers are always present |
| `readLock` / `readUnlock` | Acquire/release for readers; coordinates via `_nreaders` and `_semAccess` |
| `writeLock` / `writeUnlock` | Acquire/release for writers; directly waits/posts `_semAccess` |
