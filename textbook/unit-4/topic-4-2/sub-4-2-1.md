## What Is an Implicit Free List?

In an **implicit free list** design, the allocator has no explicit pointer linking free blocks together. Instead, to find a free block, it traverses every block in the heap — both allocated and free — using the block headers.

Because each block's header stores its size, the allocator can jump from one block to the next:

```
Next block = (char*)current_block + current_block->size
```

When it encounters a block where `allocated == 0` (free) and the block is large enough, it uses it.

## How Allocation Works

1. Start at the first block in the heap.
2. For each block: check if it is free (`allocated == 0`) AND large enough.
3. If yes: use this block (split if necessary).
4. If no: move to the next block (`block + block->size`).
5. If no block satisfies the request: call `sbrk()` for more memory.

## How Free Works

1. Mark the block as unallocated (`allocated = 0`).
2. Optionally: traverse the entire list to find adjacent free blocks and coalesce them. This is O(n) in the number of blocks.

## Drawbacks

The implicit free list has poor performance because:

- **Allocation is O(n)** in the total number of blocks, including allocated ones — most of which cannot satisfy the request.
- **Coalescing is O(n)** — finding adjacent free blocks requires a scan.

The CS252 lab does not use an implicit free list. It uses an explicit segregated free list with boundary tags to achieve O(1) coalescing and near-O(1) allocation for most sizes.

## Key Terms

| Term | Meaning |
|---|---|
| Implicit free list | No explicit free-block pointers; find free blocks by traversing all blocks |
| Block traversal | `next = (char*)current + current->size` using the size field |
| O(n) allocation | Must scan all n blocks (allocated + free) to find a free one |
