## The Problem: Finding Neighbors for Coalescing

When a block is freed, the allocator must check whether its left and right neighbors are also free so they can be merged (coalesced) into a single larger block. This is critical for reducing external fragmentation.

**Naive approach**: traverse the entire free list to find blocks that are physically adjacent to the freed block. This is **O(n)** in the number of free blocks — too slow.

**Optimized approach**: use **boundary tags** to compute neighbor addresses directly in O(1).

## The Boundary Tag Design

Every block (allocated or free) stores its own size in a **header** at the beginning of the block. The critical insight from Donald Knuth's boundary tag technique:

- To find the **right neighbor**: add the current block's size to its start address.
- To find the **left neighbor**: the header of the left block stores its own size — but you need to look it up. The CS252 implementation stores this in the current block's `left_size` field.

```
         ┌───────────────────────────────────────────────────┐
         │ Fencepost │ Block 1 (free) │ Block 2 (alloc) │ Fencepost │
         └───────────────────────────────────────────────────┘
           size=16     size=88           size=40           size=16
                       left_size=16      left_size=88
```

## Computing Neighbor Addresses

```c
// Right neighbor of 'block':
BoundaryTag *right = (BoundaryTag*)((char*)block + block->size);

// Left neighbor of 'block':
BoundaryTag *left  = (BoundaryTag*)((char*)block - block->left_size);
```

Both operations are O(1) — just pointer arithmetic.

## Why "Boundary Tag"?

The term comes from the fact that each block is "tagged" at its **boundary** with its size. The block's header serves as the tag for the right neighbor's `left_size` lookup. The system of having each block know its own size (forward pointer) and its left neighbor's size (`left_size` backward reference) is what makes O(1) neighbor lookup possible.

## O(1) Coalescing

Using boundary tags, the free algorithm can:

1. Find the right neighbor in O(1): check if it is free.
2. Find the left neighbor in O(1): check if it is free.
3. Coalesce with one or both neighbors in O(1) using their `next`/`prev` pointers.

The entire `free()` operation is O(1) (not counting the free-list reinsert, which is also O(1) since insertions go to the front).

## Slides Diagram Summary

The slides show the Boundary Tags physical layout as a flat memory strip:

```
Dummy Header | H1 (free) | Memory | H4 (free) | Memory | H3 (alloc) | Memory | H2 (alloc) | Memory | Dummy Header
```

And the free list as a separate linked structure:

```
Sentinel ↔ H4 ↔ H2 ↔ (back to sentinel)
```

Both structures coexist: the physical boundary-tag layout provides O(1) neighbor access, and the free-list pointers provide O(1) removal from the free list during coalescing.

## Key Terms

| Term | Meaning |
|---|---|
| Boundary tag | Size metadata stored at block header enabling O(1) neighbor lookup |
| `left_size` | Current block's field storing the actual size of the left neighbor |
| Right neighbor | Block at `(char*)block + block->size` |
| Left neighbor | Block at `(char*)block - block->left_size` |
| O(1) coalescing | Merging adjacent free blocks without scanning the list |
