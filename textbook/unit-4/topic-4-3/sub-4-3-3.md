## Why 8-Byte Alignment?

Many architectures require that certain data types be aligned to specific byte boundaries:

- RISC architectures: misaligned accesses cause a **SIGBUS** signal (crash).
- x86 architectures: misaligned accesses work but are significantly slower.
- `double` (8-byte floating point) must start at an 8-byte boundary.

To guarantee that any type can be safely stored in a malloc'd block, the allocator rounds all sizes up to the nearest multiple of 8.

## Rounding to 8-Byte Boundary

From the slides, the allocation algorithm step 1:

> "Round up requested size to the next 8-byte boundary (if size == 0, assume 1 byte data)"

The formula is:

```c
// Round up n to the nearest multiple of 8
size_t roundup8(size_t n) {
    return (n + 7) & ~7;
}
```

If `malloc(0)` is called, the CS252 lab returns `NULL` for determinism (per the lab spec).

## Computing the Real Size

The total actual size of a block includes both the user data (rounded up) and the header metadata:

```
actual_size = roundup8(requested_size) + sizeof(allocated_metadata)
```

From the slides:
> `real_size = roundup8(requested_size) + sizeof(header)`

## Minimum Block Size

Even though the allocated metadata is smaller (no `next`/`prev`), the block must be sized for the **full header** (including `next` and `prev`). This is because:

- When the block is **freed**, the allocator writes `next` and `prev` pointers into the block.
- If the block was not big enough to hold those pointers, freeing it would corrupt adjacent memory.

On a 64-bit system, the minimum block size is `sizeof(full_header)` where the full header includes two 8-byte pointers. This means the minimum **allocable size** is `2 * sizeof(pointer)` = 16 bytes, which is why free list index 0 (allocable sizes 1–8) is always empty.

From the lab spec:
> "The minimum actual size is the size of the full header struct. Even though the pointer fields at the end of the header are not used when the block is allocated, they are necessary when the block is free."

## Use sizeof Instead of Hard-Coded Values

A 32-bit build has different sizes than a 64-bit build. The tests include a `-m32` flag. Always use `sizeof`:

```c
// Good:
x = sizeof(size_t);

// Bad:
x = 8;   // breaks on 32-bit
```

## Allocated-Bit Packing (More Optimizations)

Since all actual block sizes are multiples of 8, the last 3 bits of the `size` field are always 0. The `allocated` flag uses only 1 bit. An optimization is to pack the allocated flag into one of those unused low bits of `size`, saving the storage for a separate `allocated` field. The CS252 lab uses a separate `allocated` field by default, but the concept is noted in the lab handout.

## Key Terms

| Term | Meaning |
|---|---|
| 8-byte alignment | All block sizes rounded to multiples of 8; required for double and RISC architectures |
| `roundup8(n)` | `(n + 7) & ~7` — rounds `n` up to next multiple of 8 |
| Minimum block size | `sizeof(full_header)` — must fit next/prev even for allocated blocks |
| SIGBUS | Signal sent on misaligned memory access on RISC architectures (SPARC) |
| Allocated bit packing | Storing allocated flag in unused low bits of the size field |
