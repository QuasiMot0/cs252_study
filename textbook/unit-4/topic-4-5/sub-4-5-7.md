## The Debugging Allocator

A **debugging allocator** adds validation checks to the standard malloc/free operations to detect memory errors early and report them with useful diagnostics. Two techniques are described in the CS252 material.

## Magic Numbers

Instead of using a single bit for the `allocated` flag, the debugging allocator uses the **entire 4-byte `allocated` field** to store a recognizable magic number:

| State | Magic Number |
|---|---|
| Free | `0xF7EEF7EE` |
| Allocated | `0xA10CA7ED` |

From the slides:
> "You can check if a block has been returned by malloc by storing a magic number in the header and checking it during free to prevent wild frees."
> "Instead of only using one bit in the flags, you could use the entire 4 bytes to store a magic number to indicate if the object is free or not."

When `free(ptr)` is called, the allocator first checks:

```c
BoundaryTag *block = (BoundaryTag*)((char*)ptr - sizeof(allocated_metadata));

if (block->allocated == 0xF7EEF7EE) {
    fprintf(stderr, "Double free detected!\n");
    abort();
}
if (block->allocated != 0xA10CA7ED) {
    fprintf(stderr, "Wild free detected! Not a malloc pointer.\n");
    abort();
}
```

After freeing, set `block->allocated = 0xF7EEF7EE`.

## heapCheck() Function

From the slides:
> "You can implement a heapCheck function that checks for the sanity of the heap and prints the free and allocated blocks. You can call it during debugging before and after suspicious free/malloc calls."

The heap check function:

1. Starts from the **first block** in the heap (the fencepost at the start of the first chunk).
2. Walks forward through blocks using `next = (char*)current + current->size`.
3. For each block, prints: offset, size, left_size, allocated status.
4. Verifies that each block's `left_size` matches the `size` of the previous block.
5. Detects inconsistencies that indicate corruption.

```c
void heapCheck() {
    // Start at first block header after initial fencepost
    BoundaryTag *block = /* first block */;
    while (/* not at end of heap */) {
        printf("offset: %p  size: %zu  left_size: %zu  allocated: %d\n",
               block, block->size, block->left_size, block->allocated);
        block = (BoundaryTag*)((char*)block + block->size);
    }
}
```

## Handling Gaps for heapCheck

When the OS has been called multiple times and the chunks are non-adjacent (holes in the heap), heapCheck needs to skip across those gaps. From the slides:

> "To get around gaps in the heap, you can store the correct size of the gaps in the fence posts around the gaps."

By recording the gap size in the fencepost headers, heapCheck can jump over gaps without reading invalid memory.

## gdb Integration

From the lab spec:
> "To debug a specific test try: `(gdb) break main` then `(gdb) run`"

Calling `heapCheck()` at strategic points (before and after suspicious `malloc`/`free` calls) and examining output in gdb helps pinpoint the location where heap corruption begins.

## Why These Tools Are Necessary

From the slides: memory allocation errors are difficult to debug because the **effect may happen far from the cause** (the crash from a double-free or premature-free may occur many calls later). The debugger shows the crash site, not the source of the corruption. Magic numbers and heapCheck narrow the search to the actual corrupting call.

## Key Terms

| Term | Meaning |
|---|---|
| Magic number | 4-byte sentinel in the `allocated` field: `0xF7EEF7EE` (free), `0xA10CA7ED` (allocated) |
| Wild free detection | On `free()`, check `allocated == 0xA10CA7ED`; if not, the pointer was not from malloc |
| Double free detection | On `free()`, check `allocated != 0xF7EEF7EE`; if it is, the block is already free |
| `heapCheck()` | Walks all blocks and validates metadata consistency; call before/after suspicious operations |
| Gap in heap | Non-contiguous chunks from multiple `sbrk()` calls; gaps stored in fenceposts for heapCheck traversal |
