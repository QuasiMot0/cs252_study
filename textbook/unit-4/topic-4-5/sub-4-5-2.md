## What Is a Memory Leak?

From the slides:
> "Memory leaks are objects in memory that are no longer in use by the program but that **are not freed**."

A memory leak occurs when a program allocates memory but loses all references to it, making it impossible to free.

## Examples

```c
// Example 1: Loop with no free — leaks 100 bytes per iteration
while (1) {
    ptr = malloc(100);
    // ptr is overwritten next iteration; old allocation is lost
}

// Example 2: Overwriting a pointer without freeing the old object
a = new Object();
// ... use a ...
a = new Object();   // old object is now unreachable and leaked
```

## Impact on Long-Lived Processes

Memory leaks are described as a **"slow but persistent problem"**:

- The heap grows continuously over time.
- Eventually, the program exhausts physical memory and begins **swapping** to disk, severely slowing the system.
- If swap space is exhausted, the system may become unresponsive.
- 24/7 server programs often need to be **"rebounced"** (shutdown and restarted) periodically because memory leaks make them too slow.

For **short-lived** programs, memory leaks don't matter in practice: the OS reclaims all memory when the process exits.

## Why Explicit Memory Management Is Leak-Prone

With languages like C/C++ that require manual `free()` / `delete`, the programmer must remember to free every allocation. Missing one free in a rarely-executed code path (e.g., error handling) is a common source of leaks.

In contrast, **garbage-collected** languages (Java, Python, C#) automatically reclaim unreachable memory, preventing leaks at the cost of runtime overhead.

From the slides:
> "Explicit Memory Allocation (calling free) uses less memory and is faster than Implicit Memory Allocation (GC). However, Explicit Memory Allocation is Error Prone."

## Detection Tools

From the slides, open-source tools for detecting memory leaks:

- **Valgrind/Memcheck** — instruments program execution to track every allocation/free.
- **Dr. Memory** — similar to Valgrind, works on Windows and Linux.
- **Clang/GCC Address Sanitizer** (ASan) — compiler instrumentation, fast overhead.
- **UndefinedBehaviorSanitizer** (UBSan) — also detects other undefined behavior.

Memory leaks are the **least critical** of the five memory errors (vs. premature frees, double frees, wild frees, and memory smashing) because the effect takes longer to manifest.

## Key Terms

| Term | Meaning |
|---|---|
| Memory leak | Allocated memory no longer in use but not freed; reference lost |
| Rebouncing | Shutting down and restarting a long-lived server to recover leaked memory |
| Valgrind | Tool that detects leaks and other memory errors by instrumentation |
| Explicit allocation | C/C++ style: programmer calls free(); fast but error-prone |
| Implicit allocation | GC style: runtime frees unreachable objects; safer but slower |
