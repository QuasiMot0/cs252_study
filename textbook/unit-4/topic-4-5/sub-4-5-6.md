## What Is Memory Smashing?

From the slides:
> "Memory Smashing happens when less memory is allocated than the memory that will be used."

It is also called **Heap Buffer Overflow** or **Buffer Overflow** memory error. The program writes past the end of an allocated block, overwriting memory that belongs to the adjacent block's header.

## How It Corrupts the Heap

Because malloc blocks are laid out contiguously in memory, the bytes immediately following the user data in one block are the **header** of the next block. Writing past the end of an allocation overwrites those header fields (`size`, `left_size`, `allocated`, `next`, `prev`).

```
Block A (allocated, size=8):         Block B (free):
┌───────────┬────────────┐           ┌────────────────────────────┐
│ A's header│  A's data  │→→→→→→→→→→│ B's header corrupted!      │
└───────────┴────────────┘           └────────────────────────────┘
                  ↑
             8 bytes allocated, but program writes 28 bytes here
```

## Example (from slides)

```c
char *s = malloc(8);
strcpy(s, "hello world cs252 students");  // 27 chars + null = 28 bytes
// We allocated 8 bytes but wrote 28
// Bytes 9-28 overwrite the header (and possibly next/prev) of the
// block immediately following s.
// Future malloc/free calls will crash.
```

The slides note:
> "Strcpy will overwrite the header and maybe next/prev of the object that comes after s, causing future calls to malloc/free to crash. Special care should be taken to also allocate space for the null character at the end of strings."

## Delayed Crash

Sometimes the smashed bytes fall in the **unused padding** region of a block (due to 8-byte alignment rounding). In that case:

- The adjacent header is not immediately corrupted.
- The error appears only later, when the adjacent block is freed or reallocated.
- This makes memory smashing especially difficult to debug.

## Common Causes

- Forgetting the null terminator: `malloc(strlen(s))` instead of `malloc(strlen(s) + 1)`.
- Using `strcpy` / `sprintf` into a fixed-size buffer without length checking.
- Off-by-one errors in loop bounds when filling an array.

## Why It's a Security Issue

Heap buffer overflows are a classic exploitation technique. An attacker who can control what is written past the end of a buffer can overwrite `next`/`prev` pointers in an adjacent free block, redirecting future allocations to arbitrary memory addresses.

## Key Terms

| Term | Meaning |
|---|---|
| Memory smashing | Writing past the end of an allocated block; also called heap buffer overflow |
| Adjacent block header corruption | The bytes past the end of one block are the header of the next block |
| Delayed crash | Smashing padding bytes causes no immediate crash; error shows up later |
| Null terminator | Off-by-one error forgetting `'\0'` is a common cause of memory smashing |
