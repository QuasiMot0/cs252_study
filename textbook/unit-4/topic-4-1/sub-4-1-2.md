## The sbrk() System Call

```c
void *sbrk(intptr_t increment);
```

- Called with a **positive** `increment`: moves the program break upward by `increment` bytes, extending the heap. Returns a pointer to the **old** break (the start of the newly allocated memory).
- Called with a **negative** `increment`: shrinks the heap.
- Called with `0`: returns the current break without changing anything.

On failure, `sbrk()` returns `(void*)-1`.

## Why the Return Value Is the Old Break

When `sbrk(s)` succeeds, it returns the old break — which is the starting address of the newly added memory. The new memory occupies addresses `[old_break, old_break + s)`.

```
Before sbrk(4096):        After sbrk(4096):
  ┌──────────────┐          ┌──────────────┐
  │  heap (used) │          │  heap (used) │
  ├──────────────┤ ← break  ├──────────────┤
  │  (unmapped)  │          │  new memory  │ ← returned pointer
  │              │          ├──────────────┤ ← new break
```

## The Ordering Constraint

`sbrk()` can only deallocate memory **in reverse order of allocation**. To return memory obtained via `sbrk()` to the OS, you must shrink the break — but that only reclaims the topmost memory. You cannot return a region in the middle of the heap while keeping memory above it.

Example:

```c
int *big_array = (int*) malloc(BIG_NUMBER);
// ...
char *name = malloc(32);
// ...
free(big_array);   // Cannot return to OS yet — name is above big_array in heap
```

Because `name` is allocated above `big_array`, the program break cannot be moved below `name`. The allocator keeps `big_array`'s memory on its internal free list rather than returning it to the OS.

## Why sbrk() Alone Is Not Enough

`sbrk()` provides only a coarse-grained primitive — it extends or shrinks the heap as a whole. To support arbitrary `malloc()/free()` in any order, the allocator must:

1. Request large chunks from the OS via `sbrk()`.
2. Manage those chunks internally with **free lists** and **block metadata**.
3. Reuse freed blocks for future allocations instead of always calling `sbrk()`.

The CS252 lab allocator requests chunks of size `ARENA_SIZE` from the OS and manages them with boundary tags and segregated free lists.

## Key Terms

| Term | Meaning |
|---|---|
| `sbrk(n)` | Extend heap by `n` bytes; returns pointer to old break |
| Program break | The end of the heap; `sbrk()` moves this boundary |
| Ordering constraint | `sbrk()` can only deallocate from the top of the heap |
| `ARENA_SIZE` | The chunk size requested from the OS at once (1KB in tests) |
