## Program Memory Sections

A running program's address space is divided into distinct regions:

| Section | Contents | Direction |
|---|---|---|
| Text | Executable instructions | Fixed |
| Data | Initialized global variables | Fixed |
| BSS | Uninitialized global variables (zeroed) | Fixed |
| Heap | Dynamic allocations (malloc/new) | Grows **upward** |
| Stack | Local variables and return addresses | Grows **downward** |

The heap sits above the BSS/Data/Text sections and grows upward (toward higher addresses). The stack sits at the top of the address space and grows downward. There is a gap between them.

## Where the Heap Lives

```
High address
  ┌─────────────────┐
  │      Stack      │  ↓ grows downward
  │                 │
  │   (free space)  │
  │                 │
  │      Heap       │  ↑ grows upward
  ├─────────────────┤
  │      BSS        │  uninitialized globals
  ├─────────────────┤
  │      Data       │  initialized globals
  ├─────────────────┤
  │      Text       │  program instructions
Low address
```

## Why This Matters for Malloc

`malloc()` must obtain memory from the OS to service allocation requests. The interface for doing this is `sbrk()`, which adjusts the **break** — the boundary between the heap and the unmapped region above it. Calling `sbrk()` with a positive value moves the break upward, extending the heap.

## Key Terms

| Term | Meaning |
|---|---|
| Text | Program instructions (read-only) |
| Data | Initialized global variables |
| BSS | Uninitialized global variables, zeroed by OS at startup |
| Heap | Region used by malloc/new; grows upward |
| Stack | Local variables and call frames; grows downward |
| Program break | Boundary between heap and unmapped memory; moved by `sbrk()` |
