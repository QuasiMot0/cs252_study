## Memory Gaps

Between the named sections (text, data, BSS, heap, stack) there are large **unmapped regions** — virtual address ranges that have no backing physical memory. These gaps exist for several reasons:

- The sections are not contiguous: heap grows up from one place; stack grows down from another.
- Not all of the 2^64 address range is ever mapped.

From the slides:
> "Memory gaps: between sections, no mapping; accessing gap → SEGV → core dump."

## SEGV — Segmentation Fault

When a program dereferences a pointer into an unmapped region, the MMU cannot translate the virtual address and raises a hardware fault. The OS converts this into a **SIGSEGV signal** sent to the process.

Common causes:
- Null pointer dereference (`int *p = NULL; *p = 5;`)
- Uninitialized pointer (`char *s; strcpy(s, "hello");`)
- Wild pointer pointing into a gap between sections
- Stack overflow (growing the stack past its limit)

The default action for SIGSEGV is to terminate the process and write a **core dump**.

## Core Dumps

A core dump is a snapshot of the process's memory at the time of the crash. It is saved to a file (typically named `core` or `core.<pid>`).

From the slides:
> "`gdb program-name core` then `gdb> where`"

The `gdb` debugger can load the core dump and show the exact call stack at the moment of the crash:

```
$ gdb ./myprogram core
(gdb) where
#0  0x0000000000401234 in strcpy (...)
#1  0x0000000000401198 in main ()
```

This reveals the crash site even without a live process.

## Key Terms

| Term | Meaning |
|---|---|
| Memory gap | Unmapped virtual address range between sections; no physical backing |
| SIGSEGV | Signal sent to process when it accesses an unmapped address |
| Segmentation fault | Common name for a SIGSEGV event; process is terminated |
| Core dump | Snapshot of process memory written to disk on crash; loadable by gdb |
| `gdb program core` | Loads a core dump; `where` command shows the call stack at crash |
