## What Is an Address Space?

Each process has its own **virtual address space** — a private, independent view of memory from address 0 to 2^64-1. The OS and CPU (via the MMU) translate virtual addresses to physical RAM addresses transparently.

From the slides:
> "Each program has its own 'Address Space'. It is independent. Modifying your own address space doesn't affect others."

## Process Isolation

Because each process has its own address space:

- A bug in one process (wild pointer, buffer overflow) cannot corrupt another process's memory.
- Two processes can both have a variable at virtual address `0x7fff5000` — they map to different physical memory locations.
- A process crash (SEGV) terminates only that process; the OS and other processes keep running.

## Shared Libraries Are the Exception

Dynamic libraries (`.so` / `.dylib` files) are **shared across processes** — the OS maps the same physical pages into multiple address spaces. This saves RAM and allows the OS to keep only one copy of commonly used code (like `libc`) in physical memory.

From the slides:
> "Dynamic libraries: shared across processes; each has own text/data/bss; one instance per system."

Each process gets its own copy of the library's **data** section (for per-process state), but the **text** (code) pages are shared read-only.

## Key Terms

| Term | Meaning |
|---|---|
| Virtual address space | Private per-process view of memory from 0 to 2^64-1 |
| MMU | Memory Management Unit; hardware that translates virtual to physical addresses |
| Process isolation | Each process's address space is independent; one crash doesn't affect others |
| Shared library | `.so`/`.dylib`; text section shared across all processes that use it; data section is per-process |
