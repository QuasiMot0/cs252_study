## What Is Post-Mortem Debugging?

From the slides:
> "This is also called 'postmortem debugging'."

When a program crashes (e.g., with a segmentation fault), the OS writes a **core file** — a snapshot of the program's memory, registers, and call stack at the moment of the crash. gdb can load this file to inspect the crash site without re-running the program.

## The Core File

From the slides:
> "When a program crashes, it writes a core file."
> "The core is a file that contains a snapshot of the program at the time of the crash. That includes what function the program was running."

```bash
bash-4.1$ ./hello
Segmentation Fault (core dumped)
bash-4.1$
```

## Enabling Core File Generation

System administrators sometimes disable core files to save disk space. From the slides:

```bash
ulimit -a                  # check current limits
# look for: core file size  (blocks, -c)  0
# If 0, enable with:
ulimit -c 1000000          # allow large core files
```

## Loading a Core File in gdb

```bash
gdb program core
```

Example session from the slides:
```
bash-4.1$ gdb hello core
GNU gdb 6.6
Program terminated with signal 11, Segmentation fault.
#0  0x000106cc in main () at hello.c:11
11       *s2 = 9;
(gdb)
```

gdb immediately shows the line where the crash occurred.

## Finding the Crash Location

```
(gdb) where
#0  0x000106cc in main () at hello.c:11
(gdb) print s2
$1 = 0x0
(gdb)
```

From the slides:
> "Now you can type *where* to find out where the program crashed and the value of the variables at the time of the crash."
> "This tells you why your program crashed."

In this example, `s2 == 0x0` (NULL) — dereferencing it caused the SEGV.

## `where` / `bt` — The Backtrace Command

`where` (alias `bt`) prints the entire call stack at the crash site:

```
(gdb) where
#0  some_function () at file.c:42
#1  another_function () at file.c:88
#2  main () at file.c:11
```

Each frame (`#0`, `#1`, ...) is one level of the call chain. Frame `#0` is where execution was when the crash happened.

## Workflow Summary

1. Compile with `-g`: `gcc -g -o hello hello.c`
2. Run until crash: `./hello` → "Segmentation Fault (core dumped)"
3. Load core: `gdb hello core`
4. Find crash: `(gdb) where`
5. Inspect state: `(gdb) print variable_name`

## Key Terms

| Term | Meaning |
|---|---|
| Post-mortem debugging | Debugging a program after it has crashed, using a core file |
| Core file | Snapshot of process memory, registers, and call stack written on crash |
| `ulimit -c 1000000` | Enables core file generation (if disabled by sysadmin) |
| `gdb program core` | Launches gdb with the crashed program and its core file |
| `where` / `bt` | Print the call stack (backtrace) at the crash site |
| Frame `#0` | The innermost stack frame — where the crash actually occurred |
